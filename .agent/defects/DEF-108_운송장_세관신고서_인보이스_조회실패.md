# DEF-108: createorder 성공 후 운송장/세관신고서/인보이스 문서 조회가 실패함

## 발견 경위
JSJung이 2026-07-18 오더 ZEN-2026-000001의 UPS createorder 성공 확인 후, "무역서류 관리"에서 운송장/세관신고서/인보이스 문서 조회 시 오류 발생을 보고.

## 조사 결과

### `INVOICE` 문서 타입 매핑 — 정상 확인
`INVOICE: '3'`(SHXK `lable_content_type=3`, "배송물류") 매핑은 JSJung 확인 결과 **의도된 정상 매핑**임(SHXK 체계에서 "배송물류" 문서가 실질적으로 인보이스 역할) — 수정 대상 아님.

### (유력 후보) `reference_no`의 하이픈
`fetchShxkTradeDocument`(443행)가 `getnewlabel`의 `listorder[].reference_no`에 `label.reference_no`(= `order.order_no`, 예: `"ZEN-2026-000001"`, 하이픈 포함)를 그대로 전달함. 어제 `cargovolume[].child_number`(패키지 참조값)가 하이픈을 거부한 전례(DEF-107)가 있고, 운송장/세관신고서/인보이스 **세 문서 조회가 전부 동시에 실패**하는 것으로 보아 문서 타입별 개별 원인보다는 **세 요청 모두가 공유하는 `reference_no` 필드 자체의 문제**일 가능성이 높음. SHXK 스펙(`docs/80_RawData/Phase8_UPS_API_리서치_결과.md:237`)에는 `listorder[].reference_no`에 별도 길이/문자 제약이 명시되어 있지 않지만, 어제도 문서화되지 않은 제약이 반복 발견됐던 패턴과 일치.

**주의**: createorder의 최상위 `reference_no`(동일 값)는 이미 성공적으로 사용됨 — 즉 createorder 단계의 검증기준과 getnewlabel 단계의 검증기준이 다를 수 있음(DEF-106의 province 필드처럼 같은 값이라도 API 엔드포인트별로 다른 제약이 적용되는 사례가 이미 있었음).

## 영향 범위
"무역서류 관리"의 운송장/세관신고서/인보이스 조회 전체.

## 긴급도
High — UPS 실연동 후속 단계(문서 조회)가 막힌 상태.

## 권장 조치
`fetchShxkTradeDocument`(및 동일 패턴을 쓰는 `fetchAndSaveLabel`, `voidUpsLabel`이 호출하는 `removeorder`)에 전달하는 `reference_no`에서 하이픈을 제거 — DEF-107과 동일한 `.replace(/-/g, '')` 패턴. DB/UI에는 원본 `order_no`(하이픈 포함)를 그대로 유지하고, **SHXK로 나가는 시점에만** 변환. Issue #582(ResultPopup)가 이미 병합되어 있어, 수정 후 재현 테스트 시 정확한 응답 JSON을 바로 확인 가능.
