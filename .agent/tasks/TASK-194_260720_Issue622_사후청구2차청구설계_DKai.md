## [설계 옵션] (Aiden 정리, Edward 확인 대기)

1. **별도 2차 인보이스 신규 발행**: `zen_invoices`에 원 오더를 참조하는 별도 인보이스 행 생성(`metadata.source_order_id` + `adjustment_of: <원 invoice_id>` 등). 이미 발행된 인보이스는 불변 유지 — 회계 관행상 가장 안전.
2. **크레딧/데빗 노트**: 차액이 양수(추가청구)면 데빗노트, 음수(환급)면 크레딧노트로 별도 문서 발행. UI 문구 중 "화주 정산 차감/크레딧 발행" 표현과 일치하는 개념이나 신규 테이블/문서 타입 설계 필요.
3. **기존 인보이스 소급 수정**: 기존 `zen_invoices.total_amount`에 조정액을 반영. 구현은 가장 간단하나 이미 화주에게 전달된 인보이스 금액이 사후에 바뀌는 것이라 회계 관행상 지양 권장.

**Aiden 1차 권고**: 옵션 1(별도 2차 인보이스) — 기존 인보이스 불변 원칙 유지, `zen_order_costs.invoice_id`를 이 신규 인보이스로 연결하면 기존 스키마(정산 조회 로직)를 그대로 재사용 가능. 단, 최종 결정은 Edward 확인 후 확정.

## [설계 확정] (Aiden, Edward 확인 완료 — 2026-07-20)

**Edward 원칙**:
1. UPS 정산서는 여러 차례 수신될 수 있으므로, 화주에게 **최종 인보이스 발행 전까지는** 정산서(원가 조정) 갱신·추가가 가능하다.
2. 정산이 마감되어 화주에게 최종 인보이스가 발행된 이후에는 소급 정산 불가.
3. [추가] 단, 화주가 인보이스를 거부(거절)하면 인보이스를 회수하고 재정산할 수 있다 — 세금계산서의 수정발급(원본 참조 + 회수/정정) 절차와 동일한 구조로 적용.
4. **정산 마감 트리거**: Agency 담당자가 본인 소속 화주의 Order 건에 대해 처리. System Admin도 처리 가능하나 **원칙적으로 금지, 예외적으로만 허용**(사유 기록 필요).

**Edward 추가 확인**: 1차 인보이스(`INV-YYYYMMDD-NNNN`)는 발행 시점에 이미 Order/Order Detail 화면에서 화주에게 **"정식 청구서가 아닌 예상 배송료"임을 명시적으로 공지**하고 있음(`UpsFreightEstimateSection.tsx` 등 기존 구현). 즉 1차 인보이스는 Edward 원칙 ②의 "최종 인보이스"에 해당하지 않는다 — 정산 마감 전까지는 1차 인보이스 자체도 갱신 대상.

**확정 모델** (옵션 1을 기본 골격으로 하되, "정산 마감" 개념으로 재구성 — 옵션 2/3 폐기):

- `zen_invoices`에 마감 여부 컬럼 추가(`is_finalized BOOLEAN DEFAULT false`, `finalized_at`, `finalized_by`, 예외 처리 시 `finalized_reason`).
- **마감 전(`is_finalized=false`)**: UPS 사후청구가 몇 차례 수신되든 `zen_order_costs.UPS_ACTUAL_ADJUSTMENT`를 갱신(기존 UPSERT 로직 유지)하고, 연결된 1차 인보이스 `total_amount`도 함께 재계산·갱신 — "예상 배송료" 상태이므로 소급 아님.
- **정산 마감 액션(신규)**: Agency 담당자(본인 소속 화주 Order 한정, RLS 가드) 또는 Admin(예외 — UI에 "예외 처리" 경고 노출 + `finalized_reason` 필수 입력)이 트리거. 마감 시점 `zen_order_costs` 최신 합계로 `total_amount` 확정, `is_finalized=true` 기록. 이후 자동 갱신 차단.
- **마감 후 조정 필요 시**(드문 사례, 예: UPS 재조정 통보): 원 인보이스는 불변 유지, `metadata.adjustment_of=<원 invoice_id>`로 연결된 신규 추가 인보이스 발행.
- **화주 거부 시**: 원 인보이스 `status='CANCELED'` 전환(스키마 기존 값 재사용) + `metadata.superseded_by`로 재발행 인보이스 연결 → 재정산 후 신규 인보이스 발행. `zen_invoice_history`(IMP-051)에 상태 변경 이력 기록.
