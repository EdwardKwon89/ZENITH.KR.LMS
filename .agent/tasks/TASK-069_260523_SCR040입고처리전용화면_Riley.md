## 설계 의견 (Agent 작성)

### **1. 입고 데이터 모델 및 검수 이력 관리 방안**

바코드 스캔 또는 오더번호 입력을 통한 창고 입고 검수 및 확정 기능(IMP-073)을 데이터베이스에 적재하고 관리하는 방식으로 두 가지 대안을 제안하며, 시스템 복잡도와 Blast Radius 최소화를 고려하여 **[방안 A]**를 추천합니다.

*   **방안 A: 기존 `order_status_history` 테이블의 `reason` 컬럼 활용 (추천)**
    *   **메커니즘**:
        1. 입고 검수 화면에서 검수 상태(정상/손상) 및 메모를 입력받습니다.
        2. 입고 확정 시 기존 Server Action인 `updateOrderStatus`를 호출하되, `nextStatus`를 `OrderStatus.WAREHOUSED`로 지정하고, `reason` 파라미터에 `[검수: 정상/손상] 메모 내용` 형태의 규격화된 문자열을 전달하여 `order_status_history.reason`에 함께 기록합니다.
        3. 이 시점에 `syncInventoryFromOrder` 함수가 자동 트리거되어 `zen_inventory_history` 원장에도 동일한 remarks가 전파됩니다.
    *   **장점**:
        *   추가적인 DB 마이그레이션이 필요 없어 스키마 변경 리스크가 **0%**입니다.
        *   기존 RLS 보안 정책 및 `update_order_status_atomic` RPC 트랜잭션 로직을 그대로 100% 재사용하므로, 결함 유발 가능성이 현저히 낮습니다.
        *   "오늘의 입고 이력" 목록 조회 시 `order_status_history`에서 `next_status = 'WAREHOUSED'` 조건으로 오더 정보와 함께 JOIN 조회하여 쉽게 구성 가능합니다.
    *   **단점**: 검수 상태(정상/손상 여부)가 문자열 안에 텍스트로 내포되므로, 향후 "손상 화물 통계"와 같이 컬럼 기반의 고도화된 정밀 쿼리가 필요할 때 LIKE 파싱 검색에 의존해야 합니다.

*   **방안 B: 별도 `zen_inbound_records` 신규 테이블 설계 및 마이그레이션**
    *   **메커니즘**:
        1. 신규 마이그레이션 파일을 통해 `zen_inbound_records` 테이블을 생성합니다. (구조: `id` UUID, `order_id` UUID REFERENCES `zen_orders`, `inspect_status` VARCHAR(20) - NORMAL/DAMAGED, `note` TEXT, `inspector_id` UUID REFERENCES `zen_profiles`, `created_at` TIMESTAMP)
        2. 입고 확정 시, 트랜잭션 내에서 `updateOrderStatus` 호출과 함께 `zen_inbound_records` 테이블에 개별 검수 레코드를 INSERT합니다.
    *   **장점**: 데이터가 물리적 컬럼으로 엄격하게 정규화되므로, 검수 상태별 쿼리 성능이 최적화되고 향후 사진 첨부(`photo_urls` 배열) 등 창고 관리 고도화 요구사항 확장이 매우 용이합니다.
    *   **단점**: 신규 마이그레이션 생성, RLS 정책 추가, Supabase TypeScript 타입 재생성, Repository 쿼리 연계 등 구현 복잡도와 Blast Radius가 상대적으로 높습니다.

*   **Riley 추천 의견**: 현재는 MVP 수준의 입고 검수 및 확정 화면을 빠르고 안정적으로 제공해야 하므로, 기존 시스템의 데이터 정합성을 가장 안전하게 보호할 수 있는 **[방안 A]**를 추천합니다.

### **2. 바코드 스캔 및 조회 처리 방안**

*   **방안 1: 수동 바코드 입력 필드 + 하드웨어 스캐너 연동 (추천)**
    *   **메커니즘**:
        1. 입고 화면의 바코드 입력창(`ZenInput`)에 포커스를 자동 자동 배치(`autoFocus`) 해둡니다.
        2. 현장의 하드웨어 바코드 스캐너(USB/Bluetooth)로 오더 바코드 또는 패키지 바코드를 스캔하면, 키보드 에뮬레이션을 통해 입력창에 텍스트가 자동 기입된 후 마지막에 개행 문자(`\n` / Enter)가 입력됩니다.
        3. 입력 폼의 `onSubmit` 이벤트가 Enter에 의해 트리거되면서 즉시 조회가 실행되도록 설계합니다.
    *   **장점**: 창고 현장의 표준 실무(하드웨어 스캐너 + PC 화면)에 가장 적합하며, 스캔 정확도와 처리 속도가 극도로 빠르고 안정적입니다.
    *   **단점**: 모바일/태블릿 단독 사용 환경에서 하드웨어 스캐너가 없을 경우 터치 입력이 다소 불편할 수 있습니다.

*   **방안 2: 브라우저 WebRTC 카메라 API (ZXing / html5-qrcode 등) 직접 연동**
    *   **메커니즘**: 디바이스의 웹캠/카메라에 접근하여 비디오 피드 상의 바코드 이미지를 실시간으로 분석/디코딩합니다.
    *   **장점**: 추가적인 하드웨어 장비 없이 모바일 단말기만으로 스캔 처리가 가능합니다.
    *   **단점**: 창고 현장의 어두운 조도나 카메라 포커싱 지연으로 인해 스캔 정확도와 반응 속도가 스캐너 기기 대비 매우 느려 대량 물량 처리에는 적합하지 않으며, 브라우저 권한 획득 등 UI 예외 처리 복잡도가 큽니다.

*   **Riley 추천 의견**: 대량의 물류 입고 검수를 안정적으로 수행하기 위해, 실무 표준인 **[방안 1 (수동 바코드 입력 필드 + 하드웨어 스캐너 연동)]** 방식으로 구현할 것을 권장합니다.

### **3. 오늘의 입고 이력 구현 방안**

*   `order_status_history` 테이블에서 `next_status = 'WAREHOUSED'` 이면서 `created_at`이 오늘 날짜(KST 기준)인 이력을 조회합니다.
*   조회된 이력에서 `order_id` 조인을 통해 품목 정보(오더 번호, 화주명, 아이템 요약) 및 검수 정보(`reason` 필드 파싱)를 추출하여 표(Table) 형태로 화면 하단에 동적으로 표시합니다.
