## [배경]

SNTL 회의록 요구사항: 오더 등록~출고 전 단계의 **예상운임표**와, 배송완료(UPS 청구서 수령) 후 **최종운임 확정**(부가요금 수기등록 포함)이 명확히 2단계로 구분되어야 함. 이미 구성요소 자체는 코드베이스에 존재하나, **전체 플로우가 실제로 매끄럽게 연결되는지 E2E 검증**이 필요함.

## [설계 확정 사항] (Edward 확정, 2026-07-23 — 트리거 방식)

중량 변경(입고 시 실측) 및 환율 변경(확정 시점) 모두 **별도 트리거 불필요** — 이미 실시간 조회 구조:
- 중량: `registerUpsOrder()`(`ups-labels.ts:294`)가 발송 시점에 `zen_order_packages.gross_weight`를 그때그때 조회
- 환율: `getNumericParam('EXCHANGE_RATE_USD_KRW')`가 확정 시점에 그때그때 조회

**따라서 이번 Task는 신규 트리거 개발이 아니라, 기존 3단계 파이프라인의 E2E 검증 및 발견된 갭 보완이 목적임.**

## [기존 파이프라인 구성요소] (중복 구현 금지 — 검증 대상)

1. **예상운임 (1단계, 등록~출고 전)**: `UpsFreightEstimateSection`/`UpsFreightEstimatePanel`(`src/components/orders/`), `estimateUpsFreight()`(`src/app/actions/ups/freight.ts`)
2. **실제 부가요금 등록 (2단계, 배송완료 후)**: `/admin/ups-actual-charges` 화면, `recordUpsActualCharges()`(`src/app/actions/finance/ups-actual-charges.ts`, TASK-194) — 오더별 추가 항목 여러 차례 입력 + `is_finalized` 플래그
3. **화주별 집계 (3단계)**: `/finance/daily-billing`(TASK-204, Riley 신규 구현) — `is_finalized=true` 오더만 집계 대상

## [검증 범위]

- 오더 1건을 실제로 등록 → 예상운임 표시 확인 → 입고(중량 실측 반영) → 출고/UPS등록 → 배송완료 → `ups-actual-charges`에서 실제 청구액+부가요금 입력 → `is_finalized` 처리 → `/finance/daily-billing`에 정확히 반영되는지 **전체 흐름 실제 UI로 1회 이상 실행**
- 각 단계 전환 시점에 화면상 "예상" vs "확정" 상태가 사용자에게 명확히 구분 표시되는지 확인(가독성/오인 가능성 점검)
- 입고 시 중량 수정이 실제로 최종 원가/운임에 반영되는지(캐시된 값을 쓰지 않는지) 직접 확인
