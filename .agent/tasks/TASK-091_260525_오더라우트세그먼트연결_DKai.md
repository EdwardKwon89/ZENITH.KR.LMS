## 설계 의견 (D_Kai 작성)

### 현황 분석

탐색 결과, **`zen_order_routes` bridge 테이블과 `selectRoute()` 서버 액션이 이미 존재**합니다. 현재 흐름:

1. `createOrder()` — 오더 생성 (route 정보 미포함)
2. `getRouteOptions(orderId)` — 3개 경로 옵션(COST/TIME/BALANCED) 계산 → `zen_route_options` 저장
3. `selectRoute(orderId, optionId)` — 선택 → `zen_order_routes` upsert

즉 경로 선택 자체는 이미 동작하나, `zen_orders`에서 선택된 세그먼트로 직접 접근하는 컬럼이 없어 TASK-092(캐리어 요율)·TASK-093(환적 추적)에서 불필요한 JOIN 체인(`zen_orders → zen_order_routes → zen_route_options → segments`)이 필요합니다.

### 3개 방안 비교

| 방안 | 설명 | 장점 | 단점 | 추정 공수 |
|:---:|:-----|:----|:----|:--------:|
| **A** | `zen_orders.route_option_id UUID FK` 추가 | 정규화 유지, `zen_orders → zen_route_options` 직접 JOIN | `zen_order_routes`와 중복 관계 발생 (selected_option_id 이관 필요) | 1일 |
| **A' (권장)** | `zen_orders.route_option_id UUID FK` 추가 + **기존 `zen_order_routes` 활용** | `selectRoute()` 수정 불필요. 읽기는 FK로 직접 JOIN | 쓰기 시 FK 유지 필요 (trigger 또는 `selectRoute()` 보강) | **0.5일** |
| **B** | `zen_orders.route_segments JSONB` 직접 저장 | JOIN Zero, 가장 빠른 읽기 | 비정규화 — `zen_route_options.segments`와 이중 관리, 동기화 위험 | 0.5일 |
| **C** | `zen_orders.via_port_ids UUID[]` 배열 | 단순 | 세그먼트별 carrier·cost·transit_days 정보 없음 — TASK-092/093 요구사항 미충족 | 0.3일 |

### 권장: 방안 A' — `zen_orders.route_option_id` FK 추가 + 기존 bridge 활용

이유:
1. **기존 인프라 재사용**: `zen_order_routes` + `selectRoute()`가 이미 동작 중 — `selectRoute()`에서 `zen_orders.route_option_id`도 함께 UPDATE 하는 방식으로 최소 변경
2. **TASK-092/093 요구 충족**: `zen_orders`에서 `zen_route_options.segments`로의 직접 JOIN 경로 확보 (1-hop: `route_option_id → zen_route_options.segments`)
3. **정규화 유지**: segments 원본은 `zen_route_options`에 단일 저장 — JSONB 복제 없음
4. **RLS**: `zen_orders` 기존 RLS 정책에 신규 컬럼 자동 포함 (별도 RLS 추가 불필요)

### 구현 계획

1. **DB 마이그레이션**: `ALTER TABLE zen_orders ADD COLUMN route_option_id UUID REFERENCES zen_route_options(id) ON DELETE SET NULL;`
2. **`selectRoute()` 수정**: `zen_order_routes` upsert + `zen_orders.route_option_id` UPDATE (트랜잭션)
3. **단, CREATE OR REPLACE VIEW `zen_order_route_summary`**: 자주 조회되는 경로 요약 정보(경유지명·총비용·ETA)를 미리 JOIN한 View 제공 — TASK-093에서 subquery 없이 사용 가능
4. **회귀 테스트**: 기존 226/226 유지 확인

> **비고**: `createOrder()` 자체는 수정 불필요 — 경로 선택은 오더 생성 후 독립 단계이므로 현행 유지가 적절.
