## 작업 결과

### 발견 경위
사용자(JSJung)가 실제 브라우저에서 "입고처리" 화면 사용 중 `Failed to load resource: the server responded with a status of 500` 보고. 로컬 dev 서버 로그 확인 결과 `/ko/warehouse/inbound`, `/ko/warehouse/pickup` 두 화면 모두 동일 원인으로 500 재현됨.

### 근본 원인
`order_status_history.changed_by` 컬럼은 `auth.users(id)`를 참조하는 FK다(`supabase/migrations/20260428235219_remote_schema.sql:610`). 그런데 `getTodayInboundHistory()`(`orders.ts`, IMP-073부터 존재하던 기존 결함)와 `getTodayPickupHistory()`(`warehouse.ts`, Task B에서 동일 패턴 복제)가 `operator:zen_profiles!changed_by(full_name)` 형태로 PostgREST 자동 관계 임베딩을 시도 — `zen_profiles`와의 직접 FK가 없어 PostgREST가 관계를 찾지 못하고 `PGRST200`으로 항상 실패.

### 변경 내용
- `src/app/actions/operations/orders.ts`: `attachOperatorNames()` 헬퍼 신규 추가(export) — `changed_by` id들을 모아 `zen_profiles`를 별도 조회 후 병합. `getTodayInboundHistory()`에서 임베딩 제거하고 이 헬퍼로 대체.
- `src/app/actions/operations/warehouse.ts`: `getTodayPickupHistory()` 동일 패턴 수정, `attachOperatorNames` import해 재사용.
- 전체 `zen_profiles!changed_by` 패턴 재검색 결과 이 2곳 외 추가 발생 지점 없음 확인.
