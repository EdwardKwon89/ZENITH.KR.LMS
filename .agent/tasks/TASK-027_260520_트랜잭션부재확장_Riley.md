## 배경

- `updateOrderStatus()`: 상태변경·히스토리·인벤토리·정산·알림·트래킹 8회 순차 쿼리 — 중간 실패 시 데이터 불일치 발생 가능

| 항목 | 내용 |
|:---|:---|
| 제안 방안 | **1. `updateOrderStatus()` 트랜잭션 래핑 (방식 A - RPC)**<br>- Supabase RPC `update_order_status_atomic` 함수 작성<br>- 매개변수: `p_order_id UUID`, `p_prev_status VARCHAR`, `p_next_status VARCHAR`, `p_reason TEXT`, `p_user_id UUID`<br>- 로직: (1) `zen_orders.master_order_id` 결합 체크 (존재 시 예외 발생 및 롤백), (2) 현재 상태가 `p_prev_status`와 일치하는지 체크 (Race Condition 방지), (3) `zen_orders.status` 업데이트, (4) `order_status_history` 이력 삽입<br><br>**2. `payInvoiceFromWallet()` 트랜잭션 래핑 (방식 A - RPC)**<br>- Supabase RPC `pay_invoice_from_wallet_atomic` 함수 작성<br>- 매개변수: `p_invoice_id UUID`, `p_user_id UUID`, `p_profile_id UUID`<br>- 로직: (1) `zen_invoices` 조회 `FOR UPDATE`로 결제 중복 잠금, (2) 이미 `PAID` 상태인 경우 예외 발생, (3) `zen_wallet` 조회 `FOR UPDATE`로 잔액 잠금, (4) 잔액 부족 시 예외 발생, (5) 잔액 차감 업데이트, (6) `zen_wallet_transactions` 이력 삽입, (7) `zen_invoices` 결제 정보 및 상태(PAID) 업데이트 |
| 선택 근거 | - 두 작업 모두 DB 내 다중 테이블의 상태 수정 및 이력 삽입을 동반하므로 원자성(Atomicity) 보장이 필수적입니다.<br>- 특히 지갑 결제의 경우, 개별 쿼리로 실행 시 잔액 차감 후 인보이스 업데이트 단계에서 오류(네트워크 순회 실패, 서버 장애 등)가 발생하면 심각한 자금 정합성 결함이 발생합니다.<br>- `SELECT ... FOR UPDATE` 잠금을 통한 동시성 제어와 DB 트랜잭션의 원자성을 보장하기 위해 RPC 호출 방식이 가장 안전합니다. |
| 예상 리스크 | - **테스트 모킹 호환성**: 기존 단위 테스트 코드(`tests/unit/finance/wallet.test.ts`)에서 Supabase `from().update()` 모의 호출을 검증하고 있을 경우, RPC 전환으로 인한 테스트 깨짐 발생 가능. RPC 모의 검증으로 동시 수정 보완 예정입니다.<br>- **외부 API 분리**: `updateOrderStatus()`에서 실행되는 이메일 발송(`Resend`), 알림 생성 등은 DB 트랜잭션 내부에서 처리할 수 없으므로, RPC 호출 성공 후 TS 단에서 비동기로 분리 실행해야 합니다. |
| 대안 방안 | - **방식 B (JS 수동 롤백)**: 중간 실패 시 이전 쿼리의 역동작을 실행하는 수동 롤백 코드를 작성할 수 있으나, 서버 Crash나 네트워크 복구 불능 상태에서는 롤백 자체가 실행되지 않을 수 있어 정합성 보장이 어렵습니다. |
