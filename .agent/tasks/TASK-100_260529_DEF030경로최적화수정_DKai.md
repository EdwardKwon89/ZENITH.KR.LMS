## 목표

DEF-030의 5개 서브이슈 중 ①~④를 수정한다. (⑤ RLS는 기조치 완료)

- **① 전체 운송사 비교 정보 제공**: `RoutingEngine`이 3개 winner만 반환하던 것을 전체 후보 + 추천 플래그로 변경, `RouteOptimizationSection` UI를 비교 테이블로 전환
- **② transport_mode 미필터링**: 인터페이스 → 어댑터 → 엔진 → 액션 전 계층에 `transportMode` 파라미터 전파
- **③ 비용=0**: `cargo_details={}` 의존 제거 → `zen_order_packages` 집계로 교체 (IMP-090 선행 해소)
- **④ tiebreaker 부재**: `selectCostOptimal` / `selectTimeOptimal` sort에 2차 정렬 추가
- **SCHEDULED 전이 가드**: 경로 미선택 오더는 REGISTERED→SCHEDULED 전이 불가 조건 추가

## 배경

Edward·Aiden이 DEF-030 영향도 분석 및 업무 흐름 검토를 완료하고 아래 방향을 확정했다.

- **"단기 대안" 채택**: 오더 생성(REGISTERED) → 상세 페이지에서 경로 선택 → ADMIN이 운송사 확인 후 SCHEDULED 전이
- REGISTERED = 화주 접수 요청, SCHEDULED = 플랫폼 운영사의 운송사 확정 — 도메인 의미 부합
- `selectRoute`의 `applied_by` + 상태 이력으로 두 시점 모두 추적 가능
- 멀티스텝 폼 통합(2단계)은 불필요 — 현재 단계 범위 외

```typescript
// 현재: 3개 winner만 반환
// 변경: 전체 후보 반환 + 추천 플래그
```

반환 방식: 모든 후보를 반환하되 각 후보에 `recommended_for: RouteOptionType[]` 필드 추가.
- COST winner → `recommended_for: ['COST']`
- TIME winner → `recommended_for: ['TIME']`
- BALANCED winner → `recommended_for: ['BALANCED']`
- 나머지 후보 → `recommended_for: []`

`RouteOption` 타입에 `recommended_for: RouteOptionType[]` 필드 추가.
