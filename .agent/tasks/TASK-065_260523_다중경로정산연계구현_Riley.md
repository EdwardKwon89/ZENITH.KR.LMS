## 배경

초기 요구사항(REQUIREMENTS.md 3.4 지능형 라우팅)에 다중 경로 배정 및 정산이 명시되었으나, Phase 3.3 경로 선택 구현 후 정산 연계가 누락된 채 완료 처리됨. `zen_route_options`(segments JSONB) 테이블은 구현됐으나 `zen_order_costs`가 `route_option_id`를 참조하지 않고 단일 운임만 계산.

**현황**:
- `zen_route_options.segments` JSONB 배열에 각 구간(segment)의 carrier, 구간 비용 등이 저장됨
- `zen_order_costs`는 `route_option_id` FK 없이 단일 cost_type 행으로만 구성됨
- `SettlementEngine.calculateOrderCosts()`는 경로 정보를 전혀 참조하지 않음
- `zen_order_routes`(최종 선택 경로)는 존재하나 정산 계산에 활용되지 않음

**핵심 과제**:

| 케이스 | 조건 | 처리 방식 |
|:---|:---|:---|
| 단일 carrier | 모든 segment의 carrier가 동일 | 합산하여 1건으로 처리 |
| 다중 carrier | segment마다 다른 carrier | 각 구간별 별도 비용 항목으로 분리 |

두 케이스를 **자동 판별**하여 처리하는 로직 설계가 핵심.

## [설계 의견] (Riley 작성)

지능형 라우팅(`zen_route_options` 및 `zen_order_routes`)에 따른 정산 연계(IMP-070)를 위해 아래와 같이 두 가지 설계 방안을 제시하며, 시스템의 안정성과 개발 효율성, 그리고 Blast Radius 최소화를 고려하여 **[방안 A]**를 추천합니다.

### **방안 A: 기존 `zen_order_costs` 테이블 컬럼 확장 및 1:N 구조 지원 (추천)**

이 방안은 기존 `zen_order_costs` 테이블에 `route_option_id` 및 구간(segment) 특화 필드(`carrier`, `segment_index` 등)를 추가하여, 단일/다중 carrier 여부에 따라 비용 레코드를 분할 저장하는 방식입니다.

1. **데이터베이스 스키마 변경 사항**:
   - `zen_order_costs` 테이블에 `route_option_id` (UUID, `zen_route_options.id` 외래키, `ON DELETE SET NULL`) 추가.
   - `zen_order_costs` 테이블에 `carrier` (TEXT, 구간별 실제 운송사명) 추가.
   - `zen_order_costs` 테이블에 `segment_index` (INTEGER, 구간 순서 구분용) 추가.
   - RLS 정책 및 기존 인덱스는 동일하게 유지 (기존 RLS 정책이 `order_id` 조인을 거치므로 별도 재설계 없이 권한 적용).

2. **비용 계산 로직 및 분기 처리 (`SettlementEngine.calculateOrderCosts`)**:
   - `zen_order_routes`에서 오더에 적용된 `selected_option_id`가 있는지 확인합니다.
   - **[Fallback]** 적용된 경로가 없으면 기존의 단일 `zen_rate_cards` 매칭 방식을 유지하여 `FREIGHT` 비용 1건을 산출합니다.
   - 적용된 경로 옵션(`zen_route_options`)이 존재하면 `segments` JSONB 배열을 분석합니다:
     - **단일 carrier 케이스**: 모든 세그먼트의 `carrier`가 동일한 경우.
       - 모든 세그먼트의 `cost` 합계를 산출하여 **1건의 `FREIGHT` 레코드**로 병합 저장합니다.
       - 단가 일관성을 위해 `quantity = chargeable_weight`, `unit_price = total_cost / chargeable_weight`로 산출해 저장합니다.
     - **다중 carrier 케이스**: 세그먼트마다 `carrier`가 서로 다른 경우.
       - 각 세그먼트별로 **개별 `zen_order_costs` 레코드**를 삽입합니다.
       - `cost_type`은 `'FREIGHT'`로 동일하게 지정하되, `carrier`, `segment_index`를 개별적으로 기록하여 구분하고 `unit_price = segment.cost / chargeable_weight`, `quantity = chargeable_weight` (또는 `quantity = 1`, `unit_price = segment.cost`) 형식으로 기록합니다.

3. **장단점 분석**:
   - **장점 (Blast Radius 최소화)**: 기존 인보이스 발행 로직(`InvoiceGenerator`), PDF 발행(`TaxInvoiceTemplate`, `CommercialInvoicePDF`), 그리고 프론트엔드의 정산 상세 내역 테이블이 `zen_order_costs` 테이블의 `*` 조회를 통해 자동으로 동작하므로 **기존 회계/출력 코드의 수정이 거의 불필요**합니다. 다중 carrier일 때 여러 행으로 쪼개진 FREIGHT 정보가 인보이스 청구서의 개별 청구 라인(Line Item)으로 자연스럽게 자동 출력됩니다.
   - **단점**: 단일 `cost_type = 'FREIGHT'`에 여러 carrier 행이 기록되므로, 통계 화면 등에서 그룹화할 때 carrier 필드를 감안하여 쿼리를 살짝 보완해야 합니다.

### **방안 B: `zen_order_cost_segments` 신규 자식 테이블 도입**

이 방안은 `zen_order_costs` 테이블에는 오더당 aggregated된 단일 비용 총합만 보관하고, 세부적인 세그먼트 구간 비용과 개별 캐리어 정보는 1:N 구조의 새로운 테이블 `zen_order_cost_segments`에 나누어 저장하는 정규화 방식입니다.

1. **데이터베이스 스키마 변경 사항**:
   - 신규 테이블 `zen_order_cost_segments` 생성:
     - `id` UUID PRIMARY KEY
     - `order_cost_id` UUID REFERENCES `zen_order_costs(id) ON DELETE CASCADE`
     - `carrier` TEXT, `from_port_id` TEXT, `to_port_id` TEXT, `cost` NUMERIC, `currency` TEXT
   - 해당 테이블에 대한 RLS 정책 및 권한 제어 추가 신설.

2. **비용 계산 로직**:
   - `SettlementEngine`에서 `zen_order_costs` 테이블에 전체 비용의 총합을 단일 레코드로 저장합니다.
   - 동시에 `zen_order_cost_segments` 테이블에 세부 세그먼트별로 레코드를 각각 삽입하여 보관합니다.

3. **장단점 분석**:
   - **장점**: 데이터베이스의 정규화 원칙에 충족하며, `zen_order_costs` 테이블의 스키마를 비교적 단순하게 유지할 수 있습니다.
   - **단점 (높은 복잡도)**: 청구서 생성 및 세금계산서 발행 로직(`InvoiceGenerator`), PDF 변환 유틸리티, 프론트엔드 UI 등 기존에 `zen_order_costs` 데이터만을 직접 사용하던 모든 코드 계층에서 신규 테이블 `zen_order_cost_segments`를 LEFT JOIN하여 구간 정보를 병합·렌더링하도록 **대대적인 코드 수정(높은 blast radius)이 수반**됩니다.

### **결론 및 추천**
기존 회계 마스터 파이프라인의 데이터 정합성을 안정적으로 보존하고 코드 변경의 사이드 이펙트를 최소화하기 위해 **[방안 A]**를 추천합니다. 방안 A는 기존 인보이스 시스템을 그대로 활용하면서도 단일/다중 carrier 비즈니스 요구사항을 직관적으로 수용할 수 있는 가장 실용적인 해결책입니다.

## [설계 확정] (Aiden 작성)

### 채택 방안: **방안 A** — 기존 `zen_order_costs` 컬럼 확장

Riley의 추천안을 채택합니다. Blast Radius 분석이 타당하며, 기존 인보이스·PDF 파이프라인이 다중 FREIGHT 행을 라인 아이템으로 자동 출력한다는 점이 결정적 근거입니다.

**확정 DB 스키마 변경**:
```sql
ALTER TABLE zen_order_costs
  ADD COLUMN route_option_id UUID REFERENCES zen_route_options(id) ON DELETE SET NULL,
  ADD COLUMN carrier         TEXT,
  ADD COLUMN segment_index   INTEGER;
```
- 3개 컬럼 모두 **NULLABLE** — 기존 레코드 하위 호환 보장
- `ON DELETE SET NULL` 필수 (`ON DELETE CASCADE` 금지 — 경로 삭제 시 비용 레코드 보존 원칙)

**확정 SettlementEngine 로직**:
1. `zen_order_routes`에서 `selected_option_id` 조회
2. **Fallback** 조건: `selected_option_id`가 없거나 `segments` 배열이 비어있으면 → 기존 `rate_cards` 매칭 로직 유지 (`route_option_id = NULL`, `carrier = NULL`, `segment_index = NULL`)
3. segments 파싱 후 단일/다중 판별:
   ```typescript
   const isSingleCarrier = new Set(segments.map(s => s.carrier)).size === 1;
   ```
4. **단일 carrier**: `SUM(segment.cost)` → 1건 `FREIGHT` 레코드
   - `carrier = segments[0].carrier`, `segment_index = NULL`, `route_option_id = selected_option_id`
   - `unit_price = total_cost / chargeable_weight`, `quantity = chargeable_weight`
5. **다중 carrier**: segments 순회 → 각 segment별 개별 `FREIGHT` 레코드
   - `carrier = segment.carrier`, `segment_index = i` (0-based), `route_option_id = selected_option_id`
   - `quantity = 1`, `unit_price = segment.cost` (영중량 나누기 오류 방지)

**통계 쿼리 보완 의무**:
- `cost_type = 'FREIGHT'`를 집계하는 기존 통계 쿼리 발견 시 `COALESCE(carrier, 'DIRECT')` 기준 그룹화 추가 (구현 범위 내 처리, 별도 IMP 불필요)

**착수 승인**: 🔍 → 🔄 전환. 즉시 구현 착수 가능.
