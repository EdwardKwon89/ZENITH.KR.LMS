## 1. 회원 및 인증 모듈

### 1.1 개인회원 가입 및 본인인증
개인 사용자가 SMS 인증을 통해 계정을 생성하는 흐름입니다.

```mermaid
sequenceDiagram
    actor User as 개인회원
    participant Web as 프론트엔드(Web)
    participant Auth as 인증서비스(Auth)
    participant SMS as SMS API(외부)
    participant DB as 데이터베이스(DB)

    User->>Web: 이름/휴대폰번호 입력 및 인증 요청
    Web->>Auth: 인증번호 생성 요청
    Auth->>Auth: 6자리 난수 생성
    Auth->>SMS: 인증번호 발송 요청
    SMS-->>User: SMS 인증번호 수신
    User->>Web: 인증번호 입력
    Web->>Auth: 인증 확인 및 중복 체크
    Auth->>DB: 휴대폰번호 중복 여부 조회
    DB-->>Auth: 결과 반환
    alt 중복 계성 존재
        Auth-->>Web: 가입 불가 안내 (이미 가입된 번호)
        Web-->>User: 오류 메시지 노출
    else 신규 가입 가능한 경우
        Auth-->>Web: 인증 성공 및 가입 폼 활성화
        User->>Web: 필수 정보(ID/PW/주소) 입력
        Web->>Auth: 회원 가입 처리 요청
        Auth->>DB: individual_member 레코드 생성 (grade_code='Family')
        DB-->>Auth: 성공
        Auth-->>Web: 가입 완료 안내
        Web-->>User: 메인 화면 이동
    end
```

### 1.3 개인회원 등급 승급 프로세스 [신규]
이용 실적에 따라 등급을 산정하고 관리자 승인을 거쳐 혜택을 부여하는 절차입니다.

```mermaid
sequenceDiagram
    participant Batch as 집계배치/StatsService
    actor Admin as 관리자(SCR-094)
    participant API as 백엔드API/PromotionService
    participant DB as 데이터베이스(DB)
    actor Member as 개인회원

    Note over Batch, DB: 1. 승급 대상자 추출 (주기적 실행)
    Batch->>DB: 설정 기간 내 실적(금액/횟수) 집계
    DB-->>Batch: 집계 결과 반환
    Batch->>Batch: 등급 요건(grade_master) 대조
    Batch->>DB: 승급 대상자 등록 (status='REQUESTED')
    
    Note over Admin, DB: 2. 관리자 심사 및 승인
    Admin->>API: 승합 심사 대상 목록 조회
    API->>DB: REQUESTED 상태 회원 조회
    DB-->>API: 목록 반환
    API-->>Admin: 목록 노출 (통계 데이터 포함)
    
    Admin->>API: 최종 승급 승인 (Approve)
    API->>DB: 회원 등급 및 할인율 갱신/이력 기록
    DB-->>API: 처리 완료
    API-->>Admin: 승인 완료 안내
    
    Note over API, Member: 3. 결과 반영
    API-->>Member: 등급 승강 알림 발송
    Member->>API: 내 정보 조회 (새 할인율 확인)
```

### 1.2 법인 심사 및 승인 프로세스
법인관리자의 가입 신청부터 관리자의 최종 승인까지의 흐름입니다.

```mermaid
sequenceDiagram
    actor Corp as 법인관리자
    actor Admin as 시스템관리자
    participant API as 백엔드API
    participant MinIO as 파일스토리지(MinIO)
    participant DB as 데이터베이스(DB)

    Corp->>API: 법인 정보 입력 및 서류 업로드
    API->>MinIO: 증빙 파일 저장
    API->>DB: corporate 레코드 생성 (status='PENDING')
    DB-->>API: 성공
    API-->>Corp: 가입 신청 완료 (심사 대기)
    
    Note over Admin, DB: 서류 검토 단계
    Admin->>API: 심사 대기 목록 조회
    API->>DB: PENDING 상태 법인 조회
    DB-->>API: 목록 반환
    API-->>Admin: 목록 노출
    
    Admin->>API: 승인(APPROVED) 처리
    API->>DB: 상태 갱신 및 시리얼(6자리) 부여
    DB-->>API: 업데이트 성공
    API-->>Admin: 처리 완료
    API-->>Corp: 가입 승인 알림 발송
```

## 2. 오더 및 운송 모듈

### 2.1 오더 등록 및 마스터오더 패킹 (Locking 지점)
하우스 오더 생성 단계부터 패킹을 통한 수정 제약 발생 시점까지의 흐름입니다.

```mermaid
sequenceDiagram
    actor Member as 회원
    actor Op as 운영자
    participant Order as 오더관리서스
    participant Calc as 운임계산엔진
    participant DB as 데이터베이스(DB)

    Member->>Order: 오더 정보 입력 (송수하인/화물)
    Order->>Calc: 예상 운송비 산출 요청
    Calc->>DB: 원가 및 회원 할인율 조회
    DB-->>Calc: 데이터 반환
    Calc->>Calc: 판매가 계산 (원가+이익-할인)
    Calc-->>Order: 예상 운임 반환
    Order-->>Member: 가격 노출 및 확정 버튼 활성화
    
    Member->>Order: 오더 등록 확정
    Order->>DB: orders 레코드 생성 (status='REGISTERED')
    DB-->>Order: 성공 (order_no 발급)
    
    Note over Op, DB: 마스터오더 구성 (Packing)
    Op->>Order: 미패킹 오더 목록 조회
    Order->>DB: REGISTERED 오더 필터링
    DB-->>Order: 목록 반환
    
    Op->>Order: 오더 선택 및 Packing 수행
    Order->>DB: master_order 생성 및 관련 오더 status='PACKED' 갱신
    Note right of DB: [LOCK] 이후 하우스오더 수정 불가
    DB-->>Order: 성공

### 2.2 이기종 데이터 표준화 매핑 프로세스 [신규]
외부 기관(항공사, 정부 등)으로부터 수신된 상이한 코드를 내부 표준으로 변환하는 흐름입니다.

```mermaid
sequenceDiagram
    participant Ext as 외부시스템/파일
    participant Int as 인터페이스서비스
    participant Map as MappingService
    participant DB as 데이터베이스(DB)

    Ext->>Int: 원천 데이터 수신 (예: 국가코드 'KR')
    Int->>Map: 코드 변환 요청 (source='MOFA', code='KR')
    Map->>DB: standard_code_mapping 조회
    DB-->>Map: 내부 표준 코드 반환 (internal='KOR')
    Map-->>Int: 변환 결과 반환
    Int->>DB: 내부 표준 코드로 비즈니스 데이터 저장
    DB-->>Int: 성공
```
## 3. 창고 및 트래킹 모듈

### 3.1 창고 입고 및 외부 API 트래킹 연동
화물 실물 도착 확인부터 실시간 운송 추적 데이터 동기화까지의 흐름입니다.

```mermaid
sequenceDiagram
    actor WH as 창고직원
    participant Sys as 창고관리시스템
    participant Track as 트래킹허브
    participant Ext as 외부운송사API
    participant DB as 데이터베이스(DB)

    WH->>Sys: 오더 바코드 스캔
    Sys->>DB: 오더 정보 조회
    DB-->>Sys: 데이터 반환
    WH->>Sys: 입고 확정 처리
    Sys->>DB: status='WAREHOUSED' 업데이트
    Note right of DB: [LOCK] 오더 정보 수정 완전 차단
    DB-->>Sys: 성공
    
    loop 주기적 동기화
        Sys->>Track: 최신 트래킹 정보 갱신 요청
        Track->>Ext: 운송지/상태 정보 조회 (API)
        Ext-->>Track: 결과 반환 (JSON)
        Track->>DB: tracking_info 생성 및 raw_data 저장
        DB-->>Track: 성공
    end
    
    Track-->>Sys: 대시보드 업데이트 알림
    Sys-->>WH: 최신 운송 현황 노출
```

## 4. 회계 및 청구 모듈

### 4.1 청구서 발행 및 입금 상태 자동 전환
운송 완료 건의 정산과 입금 확인에 따른 시스템 자동화 흐름입니다.

```mermaid
sequenceDiagram
    actor Admin as 회계관리자
    actor Member as 회원
    participant Fin as 회계시스템
    participant Mail as 이메일시스템
    participant DB as 데이터베이스(DB)

    Admin->>Fin: 배송완료(DELIVERED) 오더 목록 조회
    Fin->>DB: 정산 대상 필터링
    DB-->>Fin: 목록 반환
    
    Admin->>Fin: 청구서(Invoice) 일괄 발행
    Fin->>DB: invoice 및 invoice_item 레코드 생성
    DB-->>Fin: 성공
    Fin->>Mail: 청구서 PDF 첨부 및 발송
    Mail-->>Member: 청구서 수신 (이메일)
    
    Note over Member, DB: 입금 확인 단계
    Member->>Admin: 대금 입금 (계좌이체 등)
    Admin->>Fin: 입금액(paid_amount) 입력 처리
    Fin->>DB: payment 이력 생성 및 status 비교
    
    alt 입금액 < 청구총액
        Fin->>DB: invoice_status='PARTIAL' 갱신
    else 입금액 == 청구총액
        Fin->>DB: invoice_status='PAID' 갱신
    end
    
    DB-->>Fin: 업데이트 성공
    Fin-->>Admin: 반영 완료 및 세금계산서 발행 링크 제공
```
