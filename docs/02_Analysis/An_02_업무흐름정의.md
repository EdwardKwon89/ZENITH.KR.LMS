# 나. 업무 흐름 정의 (An_02_Process_Definition)

> **프로젝트:** ZENITH_LMS (SNTL 통합 물류 플랫폼)
> **문서번호:** An-02
> **작성자:** Antigravity (AI Agent)
> **작성일:** 2026-04-16
> **버전:** v1.0

본 문서는 SNTL 통합 물류 플랫폼(ZENITH_LMS)의 주요 비즈니스 프로세스를 정의합니다. 모든 흐름은 처리 단계별로 상세 기술되며, 예외 처리 및 비즈니스 제약 조건을 포함합니다.

## 1. 회원 및 인증 프로세스

사용자 계정 생성, 심사 및 상호작용 흐름을 정의합니다.

### 1.1 개인회원 가입 프로세스
개인 사용자가 본인인증을 거쳐 계정을 생성하는 절차입니다.
#### 1.1.1 가입 유형 선택 및 정보 입력
- **단계**: 사용자가 서비스 가입 페이지에서 '개인회원'을 선택하고 이름, 생년월일, 휴대폰번호를 입력합니다.
- **예외**: 입력값 형식 오류 시 실시간 유효성 검사 결과를 표시합니다.
#### 1.1.2 본인 인증 및 중복 체크
- **단계**: SMS 인증번호를 요청하며, 시스템은 동시에 동일 휴대폰번호로 가입된 기존 계정이 있는지 체크합니다.
- **판단**: 중복 계정 존재 시 '이미 가입된 번호'임을 안내하고 가입을 중단합니다.
#### 1.1.3 계정 정보 생성 및 통계 부여
- **상세**: 비밀번호(BCrypt 암호화), 이메일, 주소를 입력받아 계정을 생성합니다.
- **비즈니스 규칙**: 최초 가입 시 회원 등급은 시스템에 의해 자동으로 **'Family'**가 부여되며 상태는 **'ACTIVE'**로 설정됩니다.

### 1.2 법인회원 가입 및 심사 프로세스
법인관리자가 가입을 신청하고 관리자가 증빙 서류를 검토하여 승인하는 절차입니다.
#### 1.2.1 가입 신청 및 정보 입력
- **단계**: 사업자등록번호 및 법인 기본정보를 입력합니다. 시스템은 실시간으로 사업자번호 중복 여부를 체크합니다.
#### 1.2.2 증빙 서류 업로드
- **상세**: 사업자등록증 등 필수 증빙 서류를 업로드합니다. 파일은 보안 스토리지(MinIO)에 저장되며 DB에는 해당 경로(`attachment_path`)가 기록됩니다.
#### 1.2.3 관리자 서류 검수
- **단계**: 관리자가 전용 관리 페이지에서 업로드된 서류 원본을 열람하고 정보 일치 여부를 대조합니다.
#### 1.2.4 가입 승인 / 반려 처리
- **판단**: 서류가 적합하면 'APPROVED', 부적합하면 'REJECTED' 처리합니다. 반려 시에는 반드시 구체적인 사유를 입력해야 합니다.
#### 1.2.5 계정 활성화 및 법인 ID 발급
- **결과**: 승인 즉시 계정 상태가 **'ACTIVE'**로 전환되며, 시스템 시리얼 번호 **6자리**의 법인 전용 ID가 부여됩니다.

### 1.3 개인회원 등급 승급 프로세스 [신규]
이용 실적에 따라 등급을 산정하고 관리자 승인을 거쳐 혜택을 부여하는 절차입니다.
#### 1.3.1 이용 실적 데이터 집계
- **자동**: 시스템 설정(`GRADE_EVAL_PERIOD`)에 정의된 기간 동안 회원의 **발급 횟수**와 **누적 운송 금액**을 자동으로 집계합니다.
#### 1.3.2 승급 대상자 추출 및 리스트 제공
- **자동**: 등급 마스터(`grade_master`)의 요건을 충족하는 회원을 추출하여 관리자에게 '승급 대기' 리스트를 제공합니다.
#### 1.3.3 관리자 심사 및 최종 승인
- **단계**: 관리자가 대상자의 상세 실적을 복수 확인한 후 승인 버튼을 클릭합니다.
#### 1.3.4 등급 반영 및 할인율 적용
- **결과**: 승인 즉시 회원의 등급이 변경되며, 다음 오더 생성부터 변경된 등급의 **할인율**이 정산 엔진에 반영됩니다.

## 2. 오더 및 운송 관리 프로세스

운송 의뢰부터 마스터 오더 구성까지의 흐름을 정의합니다.

### 2.1 오더 등록 및 서비스 추가
#### 2.1.1 오더 기본 정보 생성
- **단계**: 송하인/수하인 정보, 화물 정보(HS코드, 중량, 용적 등)를 입력하여 오더를 생성합니다.
- **규정**: 오더 번호는 시스템에 의해 자동 생성됩니다.
#### 2.1.2 운송 서비스 결합 (Multi-Service)
- **상세**: 하나의 오더에 최대 4가지 서비스(AIR, SEA, CIR, CCL)를 추가할 수 있습니다.
- **비즈니스 로직**: 화물 정보(총중량, CBM)와 운송원가 마스터 정보를 결합하여 운송비를 자동 산출합니다.
- **확정 산식**: `최종 운송비 = 서비스 운송원가 + (원가 × 영업이익율%) - (원가 × 할인율%)` (+ Extra Charge 추가 가능)

### 2.2 마스터오더 구성 (Packing)
#### 2.2.1 오더 그룹화
- **단계**: 운영자가 여러 하우스 오더를 선택하여 하나의 마스터오더로 Packing 처리합니다.
- **상태 전이**: Packing이 완료된 오더는 상태가 'PACKED'로 변경되며, 이후 하우스 오더 단독으로는 **정보 수정 및 삭제가 불가능**합니다.

## 3. 창고 및 트래킹 프로세스

### 3.1 창고 입출고 흐름
#### 3.1.1 입고 확인 및 상태 갱신
- **상세**: 화물이 창고에 도착하면 바코드 스캔 등을 통해 입고를 확정합니다. 
- **제약**: '입고' 이후 처리 단계의 오더는 어떠한 경우에도 **수정 또는 삭제를 할 수 없습니다.**
#### 3.1.2 운송장 출력 및 출고
- **상세**: CIR(택배) 서비스 오더의 경우 PDF 운송장을 출력하여 부착 후 출고 처리합니다.

### 3.2 실시간 트래킹 업데이트
#### 3.2.1 외부 API 연동 및 데이터 보존
- **흐름**: 시스템이 주기적으로 외부 항공사/선사/택배사 API를 호출하여 최신 운송 상태를 갱신합니다.
- **규칙**: 외부 API 응답 원본(JSON)을 `raw_data` 필드에 보존하여 정합성 검증 근거로 삼습니다.

## 4. 회계 및 청구 프로세스

### 4.1 청구서 발행 및 수입 관리
#### 4.1.1 청구 명세 집계
- **상세**: 완료된 오더의 확정 운송비와 수동 입력된 Extra Charge를 합산하여 회원별 청구서를 발행합니다.
- **비고**: 운송비 산출 시 사용된 원가, 이익금, 할인금 상세 내역이 청구서 백데이터로 보존됩니다.
#### 4.1.2 입금 처리 및 상태 자동화
- **로직**: 입금 내역 확인 시 입금액을 입력하며, 청구 총액과 일치 여부에 따라 상태가 **PARTIAL 또는 PAID**로 자동 전환됩니다.
