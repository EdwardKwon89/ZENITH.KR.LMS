## 2. 🔴 Critical — 설계 오류 또는 누락

### 2.1 Section 321 자격 판단 로직 누락

**문제**: $800 면세(Section 321)는 모든 화물이 대상이 아님.

| 조건 | 설명 |
|:---|:---|
| **1인당 1일 $800 한도** | 동일 수취인이 같은 날 여러 화물을 받을 경우 합산 |
| **운임 포함 여부** | 상품 가격만 해당하는지 운임 포함인지 모호 (CBP 해석에 따라 다름) |
| **상업용 vs 개인용** | 상업용 수입은 Section 321 적용 제외 |

**영향**: `mail_cargo_type` 값을 자동 결정하는 로직이 없으면 면세/과세가 잘못 분류될 위험.

**조치**: LMS에서 동일 수취인·동일 도착일 기준으로 합산 금액을 계산하여 `mail_cargo_type`을 결정하는 규칙 엔진 설계 필요.

---

### 2.2 상태 전이 오류: `CUSTOMS_CLEARED` → `IN_TRANSIT`

**문제**: Section 7.5 상태 전이표에서 통관 완료 후 다시 `IN_TRANSIT`으로 회귀.

```
현재: CUSTOMS_CLEARED → IN_TRANSIT (부자연스러움)
제안: CUSTOMS_CLEARED → (READY_FOR_DELIVERY 또는 CARRIER_PICKUP) → 최종 배송
```

**영향**: 오더 추적 화면에서 통관 완료 후 "배송 중"으로 잘못 표시되어 고객 혼란 유발.

**조치**: 통관 완료부터 최종 배송사 전달까지의 중간 상태 신규 정의 필요.

---

### 2.3 IBC ↔ UPS 역할 구분 불명확

**문제**: 보고서는 "IBC/UPS"를 하나의 주체처럼 기술하나, 실무적으로 역할이 다름.

| 주체 | 역할 | LMS 연동 대상 |
|:---|:---|:---:|
| **IBC** | 관세사, 통관 대행, AMS 제출 | Manifest API, eTrack Callback |
| **UPS** | 최종 배송사, 라스트마일 | Tracking API, 배송 상태 |

**영향**: Manifest 제출 주체가 IBC인지 UPS인지에 따라 1차 예비신고의 API 호출 대상이 달라짐.

**조치**: 각 API의 제공자(IBC vs UPS)를 명확히 매핑하고 LMS가 각각에 대해 어떤 데이터를 전송/수신할지 분리하여 설계.

### 4.2 FXC-21과 FXC-30의 처리 차별화

### 4.7 신고 금액 산정 로직 (Customs Value Calculation)

**문제**: 통관 신고 시 **과세 대상 금액을 어떻게 산정하는지** 정의되지 않음.

| 구성 요소 | 포함 여부 | 근거 |
|:---|:---:|:---|
| 상품 가격 (Product Price) | ✅ 포함 | WCO 기준 과세 대상 |
| 배송비 (Shipping Cost) | ✅ 포함 | CBP 기준: 판매 가격의 일부 |
| 보험료 (Insurance) | ⚠️ 조건부 | CIF 조건 시 포함 |
| 할인 (Discount) | ⚠️ 조건부 | 객관적으로 입증 가능한 할인만 인정 |

**영향**: 금액 산정 로직이 없으면 자동 `mail_cargo_type` 결정(Section 321 $800 한도)과 관세 계산 모두 불가능.

**조치**: `zen_customs_invoices`에 `declared_value`, `value_component`(price/shipping/insurance/total) 필드 구조 정의 필요. 환율 적용 시점(발송일 기준 vs 도착일 기준)도 결정 필요.

### 4.8 통관 데이터 유효성 검증 규칙

**문제**: IBC/UPS API로 데이터를 보내기 전, LMS 단에서 어떤 검증을 수행해야 하는지 명세되지 않음.

| 필드 | 검증 규칙 | 실패 시 액션 |
|:---|:---|:---|
| `hs_code` | 6~10자리 숫자, 국가별 포맷 상이 | API 전송 전 차단, 오퍼레이터 수정 요청 |
| `consignee_tariff` | EIN(9자리) / TIN / PID 포맷 검증 | 국가별 포맷 상이하므로 유효성 검증 규칙 분리 필요 |
| `invoice_unitcharge` | 0 이상, 합계 = (단가 × 수량)과 일치 | Warning 후 강제 전송 (사소한 차이 허용) |
| `invoice_enname` | 영문 500자 이내, 특수문자 제한 | API Reject 방지를 위한 전처리 필요 |

**영향**: 검증 부재 시 API 호출 실패 → 재시도 → 지연 → 통관 기한 놓침으로 이어짐.

**조치**: 각 API 필드별 validation rule set 정의. 국가별/API별로 다른 rule set을 적용할 수 있도록 설계.

### 4.9 환율 처리 전략

**문제**: 통관 신고는 대부분 **USD 기준**이나, ZENITH LMS의 주문 금액은 **KRW** 기준.

| 시점 | 적용 환율 | 장점 | 단점 |
|:---|:---:|:---|:---|
| 발송일 기준 | 선적일 환율 고정 | 예측 가능, 회계 일관성 | 통관 시점과 환율 차이 발생 |
| 도착일 기준 | 실제 통관일 환율 | 관세청 요구사항에 부합 | 환율 변동 리스크, 사전 예측 불가 |
| IBC 제공 환율 | IBC 관세사가 지정한 환율 | 관세사 내부 정산과 일치 | LMS가 통제 불가 |

**영향**: 환율 적용 방식에 따라 Section 321 $800 한도 초과 여부가 갈릴 수 있음 (KRW→USD 환산 시점).

**조치**: IBC 관세사와 협의하여 적용 환율 기준 확정 필요. 가장 현실적인 방안은 **IBC 관세사 제공 환율 사용**.

### 4.10 세관 유형별 처리 차이 (Bonded / Direct / Post-Clearance)

**문제**: 보고서는 모든 통관이 동일한 프로세스를 따른다고 가정.

| 통관 방식 | 설명 | 타이밍 모델 영향 |
|:---|:---|:---|
| **Direct Clearance** | 도착 즉시 통관 | 3단계 모델 정상 적용 |
| **Bonded Warehouse / Deferred** | 보세구역 반입 후 추후 통관 | 2차 상세신고가 지연될 수 있음. 재고 관리 연동 필요 |
| **Post-Clearance Audit** | 통관 후 사후 심사 (한국 수출입 안전관리 우수 공인 업체) | 사후 서류 보관 의무, 심사 대응 프로세스 필요 |

**영향**: ZENITH LMS가 물류센터를 운영 중이라면 Bonded Warehouse 운영 가능성 있음. 이 경우 기존 3단계 모델 수정 필요.

**조치**: ZENITH LMS 물류 운영 모델(Bonded vs Non-Bonded) 확인 후 타이밍 모델 확정.

**문제**: 해외에서 반송된 상품이 한국으로 재입고될 때 한국 관세청 재통관 절차 고려되지 않음.

| 항목 | 내용 |
|:---|:---|
| **반송 사유 코드** | 세관 반려 / 화주 반송 / 주소 오류 / 수취인 거부 |

### 7.1 전체 API 맵 (발견된 3개 시스템)

PDF 분석 결과, IBC/UPS 연계는 **3개의 독립적인 시스템**으로 구성되어 있음:

| 시스템 | Base URL | 인증 방식 | 용도 |
|:---|:---|:---|:---|
| **UPS 연동 API (중국 포워더·shxk)** | `shxk.rtb56.com` | appToken + appKey | 주문 생성·수정·삭제, 운임 조회, 트래킹 |
| **IBC Air AMS** | `api.pactrak.com/manifest/aams` | Authority Token | AMS Manifest 제출 (대미 항공 세관 신고) |
| **IBC eTrack** | `api.pactrak.com/ibctrack` | Authority Token + account 파라미터 | 통관·배송 상태 조회 |
| **IBC Web Airbill** | `api.pactrak.com/ibcairbill/production` | Authority Token | Airwaybill 생성 |

> **D_Kai 분석 오류**: "IBC/UPS를 하나의 시스템"으로 간주. 실제로는 **중국 포워더 연동 API(shxk.rtb56.com)**와 **IBC API(미국 관세사 시스템)**가 완전히 분리되어 있음. 인증 방식·데이터 모델·API 엔드포인트가 모두 다름.

### 7.2 인증 체계 비교

| 항목 | D_Kai 분석 | 실제 명세 |
|:---|:---|:---|
| **중국 포워더(shxk) API 인증** | appToken + appKey | ✅ 일치. POST Body 파라미터로 전송 |
| **IBC API 인증** | appToken + appKey (추정) | ❌ **전혀 다름**. HEAD 요청 + `IBCCredentials` 헤더에 Base64(email\|password) 전송 → 응답 헤더 `Authority`에서 토큰 수령 |
| **토큰 만료** | 언급 없음 | ✅ IBC 토큰은 **유효 시간 제한** 있음. 만료 시 재발급 필요 |
| **IBC 토큰 재사용** | 언급 없음 | ✅ 만료 전까지 **여러 요청에 재사용 가능** |

**영향**: IBC API 연동 설계는 appToken/appKey 방식이 아닌 **Authority Service 기반**으로 재설계해야 함. LMS는 (1) 토큰 발급 → (2) 토큰 캐싱 → (3) 만료 시 재발급 플로우 구현 필요.

> **핵심 발견**: D_Kai가 분석한 `FXC-XX` 코드는 IBC eTrack API의 실제 상태 코드가 아님. 이는 **IBC Air AMS Test 페이지의 샘플 코드**일 가능성이 높음. 실제 운영 API는 위의 **AAMS Events + Disposition Codes** 조합으로 상태를 표현함.

#### 7.4.4 실제 eTrack 응답 구조

IBC eTrack API는 `events[]`, `aams_events[]`, `vendor_events[]`의 **3개 이벤트 배열**로 구성:

```
{
  "result": [{
    "ibc_track": "운송장번호",
    "status": "Not available yet",

    "aams_events": [ { "event_code": "A", "event_timestamp": "...", "event_location": "LAX", "event_description": "GATEWAYED" } ],
```

### 7.6 중국 포워더(shxk) API 연동 상세 (CreateOrder → SubmitForecast 2단계)

D_Kai 분석에서 누락된 **중국 포워더 API의 2단계 주문 프로세스**:

```
Step 1: Create Order (createorder)
  - order_status: "D" (Draft)로 생성
  - → 데이터 임시 저장, IBC 미전송
  
Step 2: Submit Forecast (submitforecast)
  - reference_no로 초안을 최종 제출
  - → IBC/UPS 시스템에 정식 접수
```

또는 `order_status: "P"`(Pre-announced)로 한 번에 생성 가능.

**영향**: D_Kai의 3단계 모델에서 "1차 예비신고"를 단일 API 호출로 가정했으나, 실제로는 **초안→제출 2단계** 또는 **직접 제출 1단계** 중 선택 가능. LMS 설계 시 이 옵션을 고려해야 함.
