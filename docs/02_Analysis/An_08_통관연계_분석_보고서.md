## 1. 개요

본 문서는 IBC(International Bonded Courier)와 연동 규격서를 기반으로 ZENITH_LMS의 **통관 연계 설계 상세화 가능 여부**를 분석하고, 업계 표준 사례를 조사하여 **통관 신고 시점**과 **반송/재신고 프로세스**를 제안합니다.

> **v1.1 업데이트**: An_09(통관연계 분석 검토 보고서)의 API 실사 결과를 반영하여 상태 코드 체계·인증 방식·Manifest 처리 등이 보강되었습니다. 실사 결과 PDF 1페이지에 명시된 7개 URL을 직접 Fetch하여 **3개의 독립적인 시스템**(중국 포워더(shxk)·IBC Air AMS·IBC eTrack)을 식별하였습니다.

### 1.1 분석 대상

| 출처 | 시스템 | URL | 내용 | 확보 데이터 |
|:---|:---|:---|:---|:---:|

## 3. 세관 상태 코드 체계 (IBC eTrack API 기반)

> **v1.1 업데이트**: An_09 API 실사 결과, 기존 FXC 코드(FXC-00~FXC-99)는 IBC 실운영 API에 존재하지 않습니다. 실제 체계는 **AAMS Events 4개** + **Disposition Codes 84개**의 2계층 구조로 확인되었습니다. FXC 코드는 IBC AMS Test 페이지의 샘플 데이터이거나 다른 출처일 수 있으며, D_Kai 확인이 필요합니다.

### 3.1 AAMS Events (세관 심사 상태)

IBC Air AMS 시스템이 반환하는 최상위 세관 상태입니다. eTrack API의 `aams_events[]` 배열로 수신됩니다.

| 코드 | 의미 | 설명 | 대응 액션 |
|:---|:---|:---|:---|
| `ARRIVED_AT_CUSTOMS` | 세관 도착 | 화물이 CBP 세관에 도착, 심사 대기 | 대기 |
| `CLEARED_AT_CUSTOMS` | 통관 완료 | CBP 심사 통과, 배송사 전달 준비 | 배송 상태로 전이 |
| `HELD_AT_CUSTOMS` | 세관 보류 | 추가 서류·검사 필요 | Disposition Codes 확인 후 대응 |
| `DISPOSITIONED_IN_THE_CLEARANCE_PROCESS` | 처분 결정 | 통관 과정에서 최종 처분 결정(반송·폐기 등) | Disposition Codes 확인 |

### 3.2 Disposition Codes (세부 처분 상태 — 84개)

IBC Disposition Codes API(`/v1/res/dispositioncodes`)는 84개 코드를 반환합니다. 주요 카테고리별 대표 코드는 아래와 같습니다:

| 코드 | 설명 | 카테고리 |
|:---|:---|:---|
| `OF` | Out for Delivery | 배송 |
| `DL` | Delivered | 배송 완료 |
| `HL` | Held at Location | 보류 |
| `RT` | Return to Origin | 반송 |
| `CD` | Clearance Delay | 통관 지연 |
| `IC` | In Clearance | 통관 진행 중 |
| `AF` | Arrived at Facility | 시설 도착 |
| `AD` | Additional Documents Required | 서류 보완 필요 |
| `IN` | Inspection Required | 검사 필요 |
| `EX` | Exam — Physical Inspection | 실물 검사 |
| `DE` | Destroyed | 폐기 |
| `RE` | Released by Customs | 세관 방출 |
| `RR` | Released by Regulating Agency | 규제 기관 방출 |
| `CC` | Customs Clearance Completed | 통관 완료 |
| `CA` | Carrier Acquired | 운송사 인수 |
| `DP` | Customs Penalty | 세관 벌과금 |
| `SE` | Seized by Customs | 세관 압수 |

> 전체 84개 코드 목록은 `api.pactrak.com/v1/res/dispositioncodes` 참조.  
> `HELD_AT_CUSTOMS` 이후에는 Disposition Codes를 조회하여 구체적 사유 파악 가능.

### 3.3 eTrack 3계층 이벤트 배열

eTrack API는 단일 이벤트 스트림이 아닌 **3개의 병렬 이벤트 배열**을 반환합니다:

| 배열 | 용도 | 비고 |
|:---|:---|:---|
| `events[]` | 일반 트래킹 이벤트 | 배송 단계별 기본 이벤트 |
| `aams_events[]` | 세관 심사 이벤트 | AAMS Events 4개 코드 |
| `vendor_events[]` | 운송사(Vendor) 이벤트 | 최종 배송사(DHL/UPS/USPS 등) 단계 |

**설계 영향**: 각 배열을 독립적으로 Polling하고, 최신 이벤트를 기준으로 통관 상태를 종합 판단해야 함.

## 4. 통관 신고 시점 제안

국제 특송(Express/Bonded Courier) 업계 표준 사례를 기반으로 **3단계 혼합 모델**을 제안합니다.

### 4.1 제안 모델

```
[화주] → [ZENITH_LMS] → [IBC/UPS] → [세관] → [배송]
```

| 단계 | 시점 | 수행 주체 | 내용 | 근거 |
|:---:|:---|:---|:---|:---|
| **0차 주문 생성** | 화주 주문 확정 시 | ZENITH_LMS → 중국 포워더(shxk) | CreateOrder API 호출 → Draft 저장, 이후 SubmitForecast로 운송 예약 확정 | shxk API 2단계 주문 프로세스 |
| **1차 예비신고** | 출고 직후 (Ship-out) | ZENITH_LMS → IBC API | **Fullman CSV** 형식의 Manifest 이메일 제출 (MAWB/HAWB, HS Code, 중량, 건수, 수취인). **결과는 이메일로 비동기 통보** | US AMS는 항공기 출발 전 제출 의무. IBC Manifest는 async + email 기반 |
| **2차 상세신고** | 현지 도착 전 (Pre-clearance) | IBC 관세사 | 전자 통관 신고 (invoice 상세, 관세 산정, Section 321/Entry Type 84 분기). 게이트웨이별(LAX/JFK/MIA/ORD) 시간대 고려 | 통상 24~48시간 전 사전 신고. 美 ECCF(Express Customs Clearance Facility) 표준 |
| **3차 완료 확인** | 세관 심사 통과 시 | IBC → ZENITH_LMS (Polling) | eTrack API Polling으로 **3계층 이벤트 배열**(events/aams_events/vendor_events) 조회 → LMS 트래킹 업데이트 | IBC eTrack API의 `aams_events[]` + `disp_code_history` 활용 |

### 4.2 제안 근거

- IBC는 US Customs ECCF 라이선스 보유 → 사전 통관 인프라 이미 확보
- 중국 포워더(shxk) API는 CreateOrder(주문 저장) + SubmitForecast(운송 예약)의 2단계 프로세스 — Draft 단계에서 데이터 검증 후 확정
- IBC Manifest 제출은 **비동기 이메일 기반** — 결과 통보를 위한 이메일 수신/파싱 메커니즘 필요
- eTrack Polling은 게이트웨이 시간대(PT/ET/CT)에 따라 스케줄링 필요 (LAX=PT, JFK/MIA=ET, ORD=CT)

## 5. 반송/재신고 업무 프로세스 제안

국제 특송 업계의 관세 반려 처리 표준 사례를 기반으로 **4단계 프로세스**를 제안합니다.

### 5.1 전체 흐름

```
Step 1: 반려 감지 → Step 2: 원인 분류 → Step 3: 조치 → Step 4: 재신고/종결
```

### 5.2 Step 1 — 세관 반려 감지 (Detection)

IBC eTrack API의 `aams_events[]` + `disp_code_history[]`로 실시간 감지:

| AAMS Event | Disposition Code (예시) | 의미 | 탐지 방식 |
|:---|:---|:---|:---|
| `ARRIVED_AT_CUSTOMS` | — | 세관 도착, 심사 대기 | eTrack Polling (`aams_events[]`) |
| `HELD_AT_CUSTOMS` | `AD`(Additional Documents) | 세관 보류 — 서류 보완 필요 | eTrack Polling |
| `HELD_AT_CUSTOMS` | `IN`(Inspection) / `EX`(Exam) | 세관 보류 — 검사/실물 검사 필요 | eTrack Polling |
| `DISPOSITIONED_IN_THE_CLEARANCE_PROCESS` | `RT`(Return to Origin) | 통관 거부 → 반송 | eTrack Polling + Disposition Codes |
| `DISPOSITIONED_IN_THE_CLEARANCE_PROCESS` | `DP`(Penalty) / `SE`(Seized) | 벌과금/압수 | eTrack Polling + Disposition Codes |

> Disposition Codes는 84개 전체 목록을 IBC API(`/v1/res/dispositioncodes`)에서 조회 가능. 각 코드별 대응 액션은 운영 정책에 따라 LMS에서 관리.

### 5.3 Step 2 — 원인 분류 (Classification)

| 유형 | 설명 | 처리 방향 |
|:---|:---|:---|
| **A. 서류 보완** | HS Code 불일치, invoice 정보 누락, value 증빙 필요 | → 보완 후 재신고 |
| **B. 품목 제한** | 해당 국가 수입 금지 품목, 라이선스 필요 | → 반송 또는 폐기 |
| **C. 과세 이의** | 관세 평가액 불일치, 원산지 증명 필요 | → 증빙 제출 또는 이의 제기 |
| **D. 단순 지연** | 물량 폭주로 인한 일시적 보류 | → 대기 (자동 해제) |

### 5.4 Step 3 — 조치별 워크플로우 (Action Flow)

```
                    ┌─ A. 서류 보완 → ZENITH LMS에서 보완 데이터 입력
                    │                    → IBC 관세사에 재제출 → 재신고
                    │
세관 반려 감지 ────┼─ B. 품목 제한 → 화주에게 반려 통지
                    │                    → 반송(Return to Origin) 또는 폐기
                    │
                    ├─ C. 과세 이의 → 화주 확인
                    │                    → 증빙 제출 또는 관세 납부 → 재신고
                    │
                    └─ D. 단순 지연 → 자동 재조회 (24h interval)
                                         → N회 초과 시 Aiden/Kai 에스컬레이션
```

### 5.5 Step 4 — 상태 전이 다이어그램

```
[세관 반려] → 원인 분류
                ├→ A: [보완 데이터 입력] → [IBC 재신고] → [심사] → [통과] → [배송]
                ├→ B: [화주 동의] → [반송 처리] → [재입고/폐기]
                ├→ C: [화주 동의] → [증빙 제출] → [재심사] → [통과/반려]
                └→ D: [자동 재조회] → N회 실패 → [관리자 통보]
```

## 6. ZENITH_LMS 시스템 반영 제안

| 기능 | 설명 | 우선순위 |
|:---|:---|:---:|

| **Manifest 이메일 파싱** | IBC Manifest 제출 결과 이메일 수신 및 자동 파싱 fallback | P2 |

## 7. 상세 설계 시 추가 고려 사항

현재 분석(데이터 모델·상태 코드·신고 시점·반송 프로세스)을 기반으로 상세 설계를 진행할 때 아래 6개 영역을 추가로 고려해야 합니다.

---

### 7.1 API 연동 명세 (Interface Contract)

| 항목 | 검토 필요 내용 |
|:---|:---|
| **인증 체계 분리** | 중국 포워더(shxk): appToken+appKey / IBC: Authority Header(ClientId+Token Base64) — **2개 인증 모듈 별도 구현** |
| **에러 핸들링** | IBC/UPS API 타임아웃·재시도·서킷브레이커 패턴 |
| **eTrack Polling** | IBC eTrack API Polling (3계층 이벤트 배열 처리) — 게이트웨이 시간대별 스케줄링 |
| **Manifest 제출** | Fullman CSV 생성 → 이메일 전송 → 결과 이메일 파싱 fallback |
| **FTP/SFTP 대비** | PDF에 명시된 "FTP/SFTP/email 푸시" 방식에 대한 fallback 전략 |

### 7.2 DB 스키마 설계

| 테이블 | 용도 |
|:---|:---|
| `zen_customs_invoices` | 통관 신고용 invoice 데이터 (order_id 1:N) |
| `zen_customs_logs` | 세관 상태 이력 (FXC 코드 전체 변경 이력) |
| `zen_customs_attachments` | 통관 보완 서류 (PDF/이미지) |
| `zen_hs_code_master` | HS Code 마스터 (각 국가별 관세율 포함) |

### 7.3 국가별 통관 차이 대응

현재 PDF는 **US 중심**이나, ZENITH_LMS는 한국 발송·다국적 통관을 처리:

| 국가 | 특이 사항 | 설계 반영 항목 |
|:---|:---|:---|
| **미국** | Section 321 ($800 면세) / Entry Type 84 (CBP 사전 승인) | 게이트웨이 4개(LAX/JFK/MIA/ORD) 시간대 고려, ACE/AMS 사전 통관 |
| **EU** | ICS2, IOSS VAT, 관세 150EUR 면제 | VAT 번호 수집, IOSS 등록번호 |
| **한국** | 개인통관고부호 필요, 관세 150USD 이하 면세 | 개인통관고부호(PID) 수집 |
| **중국** | Cross-border E-commerce 통관, 개인 연간 한도 | 구매자 ID·플랫폼 ID 수집 |

### 7.4 통관비 정산 로직

| 항목 | 내용 |
|:---|:---|
| **통관 수수료** | IBC 관세사 수수료를 주문별/통합 청구 |
| **관세/부가세** | 화주 부담 여부 결정 (DDP vs DDU), 선납 후 정산 |
| **반송 비용** | 통관 거부 시 반송 운임·관세 환급 처리 |

### 7.5 오더 상태 연동 (상태 전이 정의)

| 기존 오더 상태 | 통관 상태 | 전이 조건 |
|:---|:---|:---|
| `IN_TRANSIT` | `CUSTOMS_PENDING` | IBC Manifest 제출 완료 (이메일 결과 수신) |
| `CUSTOMS_PENDING` | `CUSTOMS_ARRIVED` | eTrack `aams_events[]` = `ARRIVED_AT_CUSTOMS` |
| `CUSTOMS_ARRIVED` | `CUSTOMS_HELD` | eTrack `aams_events[]` = `HELD_AT_CUSTOMS` (+ Disposition Code 확인) |
| `CUSTOMS_HELD` | `CUSTOMS_PENDING` | 보완 서류 제출 후 |
| `CUSTOMS_ARRIVED` | `CUSTOMS_CLEARED` | eTrack `aams_events[]` = `CLEARED_AT_CUSTOMS` |
| `CUSTOMS_HELD` | `RETURNING` | eTrack `DISPOSITIONED_...` + `disp_code` = `RT`(Return) |
| `CUSTOMS_CLEARED` | `IN_TRANSIT` | 최종 배송사 전달 |

### 7.6 관세청 신고 데이터 연계 (G2G)

| 항목 | 검토 필요 |
|:---|:---|
| **한국 관세청 전자통관 시스템(UNI-PASS)** | API 연동 필요 시 관세사 면허 필요 |
| **미국 CBP ACE 시스템** | IBC가 대행하므로 LMS 직접 연동 불필요 |
| **EU ICS2** | 운송사(Carrier)가 제출, LMS는 데이터 제공 |
