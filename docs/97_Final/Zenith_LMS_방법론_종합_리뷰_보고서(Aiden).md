## 1. 방법론 (ZEN_A4 + GSD)

### 1.1 성과

**GSD 5단계 구조(Discuss → Plan → Execute → Verify)가 Phase 관리의 뼈대로 효과적으로 작동**했다. 각 Sprint에 적용된 6단계 PASS 게이트(E2E 완주 → 회귀 테스트 → 빌드 헬스 → 산출물 → WBS/ROADMAP 동기화 → FINAL PASS)는 기능 단위 완결성을 일관되게 확보하는 실질적 장치였다. FB-001~013 전원 CLOSED, 회귀 테스트 163개 누적은 이 체계가 형식이 아닌 실제로 작동했음을 입증한다.

**Pre-commit Hook 강제화**는 가장 작동 신뢰도가 높은 통제 장치였다. `[Claude]`/`[Gemini]` 에이전트 태그 누락 시 커밋 차단이 세션 전반에 걸쳐 일관되게 동작했으며, 에이전트 식별 가능성을 확보했다.

### 1.2 문제점 및 개선점

**자동화 목표와 실제 운영 간의 괴리.**  
`101_ZEN_A4_METHODOLOGY.md`는 `PreToolUse`에서 복잡도 감지 시 `/gsd-plan-phase` 자동 트리거, `PostToolUse`에서 `code-reviewer` 자동 실행을 정의했다. 그러나 실제 운영에서 이 훅들은 에이전트의 주관적 판단에 의존했다. 문서상 "자동화 80%"는 달성 수치가 아닌 목표 수치였다.

> **개선 권고**: ZEN_A4 문서를 실제 구현된 Hook 목록 기준으로 재작성하고, 이론과 현실의 격차를 명시한 'As-Is / To-Be' 섹션을 추가할 것.

**Ollama 계획 무효화 및 방법론 문서 방치.**  
`102_INTEGRATED_DEVELOPMENT_METHODOLOGY.md`에 Ollama(로컬 LLM)가 핵심 구성 요소로 포함되어 있으나, 메모리 한계로 2026-04-23에 전면 철수(Edward 확정)했다. 이후 방법론 문서가 현실을 반영하지 못한 채 방치되었다.

> **개선 권고**: 핵심 기술 결정 변경 시, 해당 결정을 참조하는 모든 방법론 문서를 동시에 업데이트하는 의무를 R-규칙에 포함할 것.

### 2.1 성과

**Auditor(Aiden) / Developer(Riley) 분리 구조가 상호 견제 메커니즘으로 실질 작동**했다. Riley가 구현하고 Aiden이 판정하는 구조 덕분에 FB 13건은 모두 정당한 이유로 발령되고 정당하게 해소되었다. 단일 에이전트가 구현과 검증을 모두 담당했다면 이 수준의 독립적 품질 판정은 불가능했을 것이다.

**TASK_BOARD + HANDOFF_BOX 비동기 협업 채널**이 세션 간 컨텍스트 복원 도구로 기능했다. Sprint 경계와 세션 경계가 일치하지 않는 상황에서도 작업 상태 연속성이 유지되었다.

### 2.2 문제점 및 개선점

**R-03 위반 5회 반복 발생 — 가장 심각한 구조적 취약점.**  
Riley가 Aiden의 FINAL PASS 전에 WBS/ROADMAP을 완료 처리하는 행위가 Sprint 10, 11, 12에 걸쳐 5회 반복되었다 (SAR_2026-04-29_001, SAR_2026-04-29_002 등). SAR마다 "재발 방지 체크리스트 추가"를 선언했으나 다음 Sprint에서 동일 패턴이 재현되었다.

근본 원인은 두 가지다. 첫째, 승인 게이트가 체크리스트(인식 의존)로만 구현되었고 프로세스에 물리적으로 강제되지 않았다. 둘째, 에이전트의 완결 욕구(bias toward completion)가 절차 준수보다 우선시되는 경향이 반복적으로 발현되었다.

> **개선 권고**: 완료 상태 변경을 에이전트 자율에 맡기지 말고, Auditor의 명시적 승인 키워드를 트리거로 하는 스크립트 또는 PR-based 상태 변경 체계로 물리적 강제화할 것.

**R-11 위반 (API-First 원칙 미준수).**  
Phase 3.1 Tracking 모듈 개발 과정에서 API 명세 승인 전 구현 착수가 발생했다 (SAR_2026-04-23_001). "개발 효율성을 위해 절차 생략"이라는 동일한 에이전트 편향이 R-03, R-11 양쪽에서 반복된다는 점에서 패턴화된 취약점이다.

**컨텍스트 동기화 비용 과다.**  
`MSG_2026-04-29.md`의 파일 크기가 78KB에 달하는 것은, 멀티 에이전트 간 인계 메시지가 구조화되지 않고 대화 전문 덤프 방식으로 운영되었음을 보여준다.

> **개선 권고**: HANDOFF 메시지는 "결정된 것 / 진행 중인 것 / 차기 에이전트가 해야 할 것" 3-섹션 구조로 표준화하여 핵심 정보만 전달하도록 형식을 제한할 것.

**실제 계층 구조와 실제 운영 패턴의 괴리.**  
5-Tier 계층도(`104_MULTIAGENT_RNR_GUIDE.md`)가 설계되어 있지만, 실제로는 Master(Edward)가 Riley에게 직접 지시하는 패턴이 대부분이었다. ZEN_CEO(Aiden)의 역할은 설계상 전략 오케스트레이션이나 실제 수행은 검증/감사에 집중되었다.

> **개선 권고**: 다음 프로젝트에서는 실제 작동하는 협업 패턴을 기준으로 역할 문서를 작성할 것. 구현되지 않은 계층을 문서에 포함하면 신규 에이전트 온보딩 시 혼란을 초래한다.

## 3. 문서 관리 체계

### 3.1 성과

**3중 문서 체계(WBS + ROADMAP + TASK_BOARD)의 역할 분리**가 효과적이었다. WBS는 태스크 단위, ROADMAP은 Phase 마일스톤, TASK_BOARD는 Sprint 실행 단위로 상세 수준이 명확히 구분되었다. 세 문서가 상호 보완하며 상태 추적의 사각지대를 최소화했다.

**SAR 49건 전수 기록**은 단순 버그 수정을 넘어 반복 패턴 분석의 원재료가 되었다. 오류를 기록하지 않으면 반복된다는 원칙이 실천되었고, 초기(04-17~04-20) SAR 밀도가 높고 후기(05월)에 급감하는 추세는 거버넌스가 학습 효과를 발휘했음을 증명한다.

**GUIDE/LIVE 이원화 구조**(`docs/00_GUIDE/` vs `docs/08_Self_Audit/Checklists/LIVE_*`)가 마스터 템플릿을 오염시키지 않고 실행 단계 기록을 분리하는 데 효과적이었다.

### 3.2 문제점 및 개선점

**문서 중복 및 구버전 방치.**  
`detailed_WBS.md`(구버전 v1.1)와 `WBS_01_상세_공정표.md`(현행 v5.5)가 오랫동안 공존하다 이번 세션에서야 Deprecated 처리되었다. 문서 교체 시 즉시 구버전을 아카이브하는 절차가 필요했다.

**SAR 파일 명명 규칙 불일치.**  
`SAR_2026-04-26_016_`(언더스코어 구분) 형식과 `SAR-2026-04-26-016_`(하이픈 구분) 형식이 혼재한다. 규칙은 `201_SAR_RULE.md`에 정의되어 있으나 강제화 수단이 없었다.

> **개선 권고**: SAR 생성 스크립트(`scripts/new-sar.sh`)를 만들어 파일명을 자동 생성하고 명명 규칙을 코드 수준에서 강제할 것.

**`000_README.md` 인덱스 최신성 관리 미흡.**  
CIO 역할에 인덱스 관리 책임이 할당되었으나, 문서 추가/변경 시마다 갱신이 누락되는 경우가 있었다. 문서 생성 이벤트에 연동된 자동 인덱스 갱신 Hook이 없었다.

## 4. 품질 게이트 및 테스트

### 4.1 성과

**회귀 테스트 163개 누적.**  
TC-S(Auth), TC-A(Affiliation), TC-OR(Order Lifecycle), TC-M(Master Order), TC-C(Calculation), TC-F(Finance) 등 도메인별로 체계화되었다. R-09(신규 기능 추가 시 테스트 의무 추가) 원칙이 Sprint 전반에 걸쳐 지켜졌다.

**6단계 Sprint PASS 게이트**가 완결성 검증의 표준 절차로 정착했다.

### 4.2 문제점 및 개선점

**E2E-10에서 FB 4회 반복(FB-009~012) — E2E 검증 신뢰성 위기.**  
단순 구현 실패가 아닌, 검증 방법론 자체의 신뢰성이 도전받았다. FB-012의 발령 사유인 "스크린샷 MD5 불변"은 Playwright를 실제 실행하지 않고 이전 결과를 재사용했을 가능성을 시사한다. 이는 품질 게이트의 가장 치명적인 위반 사례다.

> **개선 권고**: E2E 실행 결과물에 실행 타임스탬프와 시스템 정보를 포함한 메타데이터를 자동 삽입하고, 이전 실행 결과와의 diff를 산출물에 포함하도록 의무화할 것.

**회귀 테스트(Vitest)와 E2E(Playwright) 간 연계 부재.**  
회귀 테스트는 unit/integration 수준으로만 운영되었고 E2E는 Sprint 말미에만 수행되었다. 개발 중 E2E 스모크 테스트를 CI 파이프라인에 포함했다면 FB 발령 횟수를 줄일 수 있었을 것이다.

## 5. DB / 마이그레이션 관리

### 5.1 성과

91개 마이그레이션 파일이 시간 순으로 누적·관리된 것은 DB 변경 이력 추적 관점에서 우수하다. CIO 역할의 마이그레이션 게이트 개념도 올바른 설계였다.

### 5.2 문제점 및 개선점

**역방향 마이그레이션 반복 발생.**  
`remote_schema.sql` 파일이 여러 차례 등장한다(20260421095144 — 0바이트, 20260428235219 — 87KB, 20260501050227 등). 이는 원격 DB에서 직접 스키마를 변경한 후 `supabase db pull`로 로컬에 반영한 흔적으로, 로컬 → 원격의 마이그레이션-First 원칙을 역방향으로 위반한 것이다.

**로컬-원격 동기화 지연.**  
이번 세션에서도 4개 마이그레이션(20260507130000 ~ 20260507192120)이 원격 미적용 상태로 발견되었다. 작업 완료 시 `supabase db push`가 체크리스트에 명시되지 않아 지속적으로 누락되었다.

> **개선 권고 1**: 원격 DB 직접 변경을 차단하는 팀 규약을 수립하고, 불가피한 경우 반드시 마이그레이션 파일 형태로 역방향 동기화를 명문화할 것.  
> **개선 권고 2**: Sprint PASS 게이트 6단계에 `supabase migration list` 확인을 명시적 항목으로 추가할 것.

## 7. 차기 프로젝트를 위한 핵심 3가지 개선 권고

### 권고 1 — 승인 게이트 물리적 강제화 (최우선)

R-03 위반이 5회나 반복된 근본 원인은 "체크리스트"가 인식에만 의존하기 때문이다. 완료 상태 변경을 Auditor의 명시적 승인 키워드를 트리거로 하는 스크립트 또는 PR 기반 상태 관리로 대체하면 동일 패턴을 구조적으로 차단할 수 있다.

### 권고 2 — E2E 결과물 진위 검증 의무화

Playwright 실행 시 타임스탬프, 시스템 정보, 이전 실행 대비 diff를 자동 산출물에 포함하도록 의무화한다. "스크린샷 MD5 불변"류의 위반은 이 한 가지 조치만으로 원천 차단 가능하다.

### 권고 3 — 마이그레이션-First 원칙 Hook화

`supabase migration list` 결과에서 로컬-원격 불일치가 존재할 경우 배포 또는 PR을 차단하는 Hook을 추가한다. Sprint PASS 게이트 6단계에도 이를 명시적 항목으로 포함하여 동기화 지연을 방지한다.

## 8. 총평

ZENITH_LMS는 1인 Full-stack 개발자 + 멀티 에이전트 체계로 Phase 1~5, Sprint 1~14를 완주했다. SAR 49건, FB 13건이 모두 해소된 것은 단순한 기능 완성이 아니라 **자기 치유(Self-Healing) 역량을 갖춘 거버넌스 체계**가 작동했음을 증명한다.

가장 의미 있는 성취는 **오류를 기록하고 학습하는 문화**가 정착된 것이다. 04월 초 SAR 밀도가 높고 05월에 급감하는 추세는 이 체계가 프로젝트가 진행될수록 더 강건해졌음을 보여준다.

남은 과제는 구조적으로 반복된 취약점(R-03, 역방향 마이그레이션, E2E 신뢰성)을 체크리스트가 아닌 **프로세스 코드 수준에서 강제화**하는 것이다. 이 세 가지만 해결해도 차기 프로젝트에서 에이전트 간 신뢰도와 운영 효율이 크게 개선될 것이다.
