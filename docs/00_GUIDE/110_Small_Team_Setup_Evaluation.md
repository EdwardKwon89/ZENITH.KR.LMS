# 1~5명 팀 소프트웨어 개발: 설정 평가 & 보완안

> A++++ vs GSD+A++++ 설정의 실무 적용성 분석

**작성일:** 2026-04-06  
**대상:** 1~5명 팀 기반 프로젝트  
**목표:** 각 설정의 강점/문제점/보완안 제시  

---

## 📊 **Executive Summary**

| 설정 | 추천도 | 팀 규모 | 프로젝트 규모 |
|------|-------|--------|-------------|
| **A++++** | ⭐⭐⭐⭐ | 1~3명 | MVP ~ 중간 규모 |

## 1️⃣ A++++ 설정 분석

### ✅ 장점

#### 1. 빠른 적용
```
시간: 30분 내 설정 완료
학습곡선: 낮음 (즉시 사용 가능)
오버헤드: 최소화
```

#### 2. 운영 효율성
```
매일 업무:
  - 코드 작성 시 자동 검증 (code-reviewer)
  - 저장 시 자동 리뷰 (PostToolUse)
  - 커밋 전 최종 검증 (Stop)
  
관리 비용: 거의 없음
```

#### 3. 빠른 피드백
```
개발 → 자동 검증 → 즉시 피드백 (빠름)
대기 시간: 거의 없음
```

#### 4. 1~2명 팀에 완벽
```
- 자동화로 인한 수동 리뷰 감소
- 한 명이 여러 역할 수행 가능
- 셀프 리뷰 + 자동 리뷰 조합
```

### ❌ 문제점

#### 1. 대규모 리팩토링 추적 불가
```
문제:
  - 10개 파일 수정 → 각각 독립적 리뷰
  - 전체 맥락 파악 어려움
  - 순환 의존성 등 구조 문제 미감지

예시:
  File A 수정 → code-reviewer 통과
  File B 수정 → code-reviewer 통과
  하지만 A-B 간 의존성 문제 → 미감지
```

#### 2. 팀 협업 정보 손실
```
문제:
  - "왜 이렇게 설계했는가?"에 대한 기록 없음
  - 다른 팀원이 의도 파악 어려움
  - 의사결정 근거 미흡

예시:
  개발자 A: 특정 방식으로 구현
  개발자 B: 3주 후 작업할 때 배경 모르고 다시 수정
```

#### 3. Context Rot 위험
```
문제:
  - 장기 프로젝트(3개월+) 시 컨텍스트 손실
  - "이게 왜 이렇게 되어 있지?" 상황 증가
  
예시:
  2026-04: 기능 A 구현 (결정: 성능 우선)
  2026-07: 기능 B 추가 (맥락 모르고 다르게 설계)
  → 나중에 일관성 문제 발견
```

#### 4. 3명 이상 팀에서 오버헤드
```
문제:
  - 팀 의사결정 기록 없음
  - "누가 뭘 정했는가" 불명확
  - 회의 내용 정리 필요

예시:
  회의: "API 응답 형식을 X로 하자"
  → 기록 없음 → 나중에 혼동
```

#### 5. 다중 브랜치 충돌 관리 어려움
```
문제:
  - 3명이 동시에 작업할 때
  - Merge conflict 증가
  - 의존성 추적 어려움

시나리오:
  개발자 A: feature/auth 작업
  개발자 B: feature/api 작업
  개발자 C: feature/db 작업
  → 각각 독립적 리뷰 → 통합 시 문제 발생
```

### 🔧 보완 사항 (A++++ 사용 시)

#### 1. 문서 중심 접근
```
추가할 것:
  ├─ .planning/DECISIONS.md (아키텍처 결정)
  ├─ .planning/CONTEXT.md (공유 컨텍스트)
  └─ docs/ARCHITECTURE.md (설계 문서)

매주:
  - 팀 회의 기록
  - 주요 결정사항 정리
```

#### 2. Commit Message 템플릿
```
추가:
  - Conventional Commits 엄격히 준수
  - Body에 "왜"를 기록
  - "Fixes #123, Relates to #456" 기록

예시:
feat: Add JWT authentication

- Implemented JWT token generation
- Added token validation middleware
- Configured token expiration (24h)

**Why:** Enhanced security by replacing session-based auth
**Decision:** JWT chosen for stateless architecture
**Related:** #123 (Auth milestone)
```

#### 3. Code Review 체크리스트
```
추가할 것:
  docs/CODE-REVIEW-CHECKLIST.md
  
내용:
  - 아키텍처 일관성 확인
  - 다른 모듈과의 영향도 분석
  - 성능 영향도 검토
```

#### 4. Weekly Context 동기화
```
추가할 것:
  매주 금요일:
    - .planning/CONTEXT.md 업데이트
    - 주요 결정 정리
    - 다음주 예정 공유
```

## 2️⃣ GSD + A++++ 설정 분석

### ✅ 장점

#### 1. 완벽한 팀 협업
```
장점:
  - CONTEXT.md로 모든 의사결정 기록
  - "왜"가 명확함
  - 신규 팀원 온보딩 용이
```

#### 2. 장기 프로젝트 안정성
```
장점:
  - Context rot 방지
  - 6개월 후에도 의도 파악 가능
  - 기술부채 추적 용이
```

#### 3. 대규모 리팩토링 관리
```
장점:
  - Plan 단계에서 전체 영향도 분석
  - Execute 단계에서 Wave별 추적
  - Verify 단계에서 통합 검증
```

#### 4. 정확한 기능 추적
```
장점:
  - 각 기능이 설계 → 구현 → 검증 과정 추적
  - 190개 기능 관리도 가능
  - 기능 간 의존성 명확
```

### ❌ 문제점

#### 1. **오버헤드 과다** (1~3명 팀)
```
문제:
  - Initialize: 1-2시간 소요
  - Discuss: 1-2시간
  - Plan: 2-3시간
  - 총 4-7시간 메타작업
  
소규모 팀 입장:
  "코드 한 줄도 안 짰는데 하루를 투자했다"
```

#### 2. 빠른 프로토타입 어려움
```
문제:
  - MVP 1주일 목표
  - GSD Initialize/Discuss/Plan: 2-3일 소요
  - 실제 개발: 3-4일
  
현실:
  "우리는 빠르게 프로토타입하고 학습해야 하는데,
   계획에 시간을 다 쏟고 있다"
```

#### 3. 팀원 리소스 부족
```
문제:
  GSD 각 단계 담당자:
    - Initialize: 아키텍트 필요
    - Discuss: 전체 팀 필요
    - Plan: 기술 리드 필요
    - Execute: 개발자 필요
  
1~3명 팀:
  "모든 역할을 내가 다 해야 한다"
```

#### 4. 프로세스 오버엔지니어링
```
문제:
  - 1개 기능 구현 시에도 5단계 과정
  - 버그픽스에도 GSD 적용?
  - 사소한 문서 수정에도?
  
문제 상황:
  버그 하나 고치는데 30분 소요
  → GSD 과정 3시간
```

#### 5. 러닝커브가 가파름
```
문제:
  - GSD 개념 학습 필요
  - /gsd-new-project 등 명령어 학습
  - Plan 단계의 XML 형식 이해
  
신규 팀원:
  "1주일 훈련 받고 시작한다?"
```

### 🔧 보완 사항 (GSD+A++++ 사용 시)

#### 1. 경량화된 GSD (Lightweight GSD)
```
원본 GSD: Initialize → Discuss → Plan → Execute → Verify (5단계)

경량화 GSD (1~5명용):
  ├─ Plan만 상세 (복잡한 기능)
  ├─ Execute는 빠르게 (코드 직접 작성)
  └─ Verify는 자동화 (A++++ orchestration)

설정:
  - 간단한 기능: /gsd-quick (15분)
  - 복잡한 기능: /gsd-plan-phase (1시간)
  - 버그픽스: GSD 생략 (A++++ 직접 사용)
```

#### 2. 조건부 GSD 적용
```
언제 GSD 사용할까:

✅ GSD 필수:
  - 새로운 아키텍처 결정 필요
  - 3명 이상이 함께 작업할 부분
  - 장기 (3개월+) 프로젝트
  - 시스템 리팩토링

❌ GSD 불필요:
  - 버그 수정
  - 간단한 기능 추가 (1-2일)
  - 문서 작성
  - 의존성 업데이트
```

#### 3. 간결한 CONTEXT.md
```
원본:
  - 300줄 상세 컨텍스트 문서

경량화:
  - 30줄 핵심만 (2~3분 읽기)
  
내용:
  1. 프로젝트 목표 (2줄)
  2. 기술 스택 (5줄)
  3. 아키텍처 다이어그램 (1개)
  4. 주요 결정 (5개)
  5. 팀 정보 (3줄)
```

#### 4. 자동화된 Plan
```
수동 Plan (어려움):
  - /gsd-plan-phase 실행
  - 사용자가 여러 질문에 답변
  - 시간 소요: 1-2시간

자동화된 Plan:
  - templates/ 에서 템플릿 선택
  - 기본값으로 시작
  - 필요시만 커스터마이징
  
예시:
  /gsd-plan-phase --template simple --feature-count 5
  → 15분 내 Plan 완료
```

## 3️⃣ 1~5명 팀 시나리오별 분석

### 시나리오 A: 1명 팀 (개인 프로젝트)

```
특성:
  - 빠른 의사결정
  - 자기 코드만 리뷰
  - 문서화 선택적

권장: A++++ (90% 추천)
이유:
  ✅ 오버헤드 최소
  ✅ 즉시 시작 가능
  ✅ 자동 리뷰로 품질 보장

보완:
  - 주석 상세히 (자신을 위해)
  - Commit message 명확히
  - 월 1회 코드 정리

GSD 사용: ❌ 불필요
```

### 시나리오 B: 2~3명 팀 (스타트업)

```
특성:
  - 빠른 기능 출시 필요
  - 의사결정이 빈번
  - 팀 커뮤니케이션 부족 가능

권장: A++++ + 가벼운 GSD (70% A+++, 30% GSD)
이유:
  ✅ 대부분은 A++++ 자동화
  ✅ 중요 결정만 GSD Plan
  ✅ 밸런스 좋음

사용 규칙:
  - 일반 기능: A++++ (코드 작성 → 자동 리뷰)
  - 아키텍처 변경: /gsd-plan-phase (30분)
  - 버그: A++++ (직접 수정)
  - 팀 회의: CONTEXT.md 간단히 업데이트

GSD 사용: 선택적 (중요할 때만)
```

### 시나리오 C: 3~5명 팀 (중소 회사)

```
특성:
  - 팀원 간 작업 조율 필요
  - 기술 의사결정 논의 필요
  - 신규 팀원 온보딩 가능

권장: GSD+A++++ (50% GSD, 50% A+++)
이유:
  ✅ 팀 협업 명확화
  ✅ 의사결정 기록 유지
  ✅ 온보딩 용이

사용 규칙:
  - 주(week) 시작: /gsd-plan-phase (주간 계획)
  - 일일 작업: A++++ (코드 작성)
  - 주(week) 종료: /gsd-verify-work (검증)
  - 주요 결정: CONTEXT.md 기록

GSD 사용: 계획 단계만
```

## 4️⃣ 최종 권장: 하이브리드 설정

### 🎯 권장 구조

```
┌─────────────────────────────────────────┐
│  Lightweight Adaptive Setup             │
│  (1~5명 팀용 최적화)                     │
└─────────────────────────────────────────┘

BASE: A++++ (항상 활성화)
└─ PreToolUse: 자동 검증
└─ PostToolUse: 자동 리뷰
└─ Stop: 최종 게이트

CONDITIONAL: GSD (필요시만)
├─ 팀 크기 >= 3명 → GSD Plan 활성화
├─ 프로젝트 기간 >= 3개월 → GSD Discuss 활성화
├─ 아키텍처 변경 → /gsd-plan-phase 실행
└─ 팀 회의 → CONTEXT.md 간단히 업데이트

AUTOMATION: 최대화
├─ code-reviewer 자동 리뷰
├─ frontend-patterns 자동 검증
├─ security-reviewer 보안 체크
└─ verify 자동 완료도 확인
```

### 📋 적용 체크리스트

#### Week 1: 기본 A++++ 설정
```
□ .claude/settings.json 생성 (A++++ 기본)
□ team members 3명 이상? → GSD 추가 검토
□ 프로젝트 기간 예상? → 3개월+ → GSD 준비
□ 첫 기능 구현 (A+++ 자동화 테스트)
```

#### Week 2-4: 팀 크기에 따라 조정
```
1명 팀:
  □ A++++ 유지
  □ Commit message 가이드 작성

2~3명 팀:
  □ A++++ 계속 사용
  □ CONTEXT.md 간단히 시작 (30줄)
  □ 월 1회 코드 리뷰 미팅

4~5명 팀:
  □ GSD Plan 추가 (주간 계획)
  □ CONTEXT.md 상세히 (100줄)
  □ 주 1회 기술 의사결정 미팅
```

```
✅ 1~2명 팀
✅ MVP/프로토타입 (1~3개월)
✅ 빠른 피드백 중요

❌ 3명 이상 팀
❌ 복잡한 아키텍처
```
