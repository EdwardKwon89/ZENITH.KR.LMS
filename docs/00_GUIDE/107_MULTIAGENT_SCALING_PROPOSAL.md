## 1. 문제 제기 (Edward, 2026-07-18)

> "2일 동안은 Edward/Aiden이 작업을 하지 않고 Auditing 중심으로 작업을 수행해서, 코드 및 문서가 Conflict가 발생하지 않은 것 아닌가?"

이 지적이 정확하다. 지난 2일간 관측된 "충돌이 적었다"는 경험은 현재 구조가 확장 가능함을 증명하지 않는다 — Aiden이 능동적 개발(파일 상시 편집)이 아니라 배치 단위 읽기/병합만 수행했기 때문에 생긴 착시일 가능성이 높다. 15인 규모의 동시 개발(별도 신규 프로젝트 예정)을 앞두고, 이 착시를 걷어내고 실제 확장 한계를 짚어야 한다.

## 2. 핵심 진단 — 이미 3인 규모에서도 균열이 있었다

Team B는 3명(Dave·Baker·Mike)뿐이었는데도, `.agent/VIOLATION_TRACKER.md` 실측 기록상 아래와 같은 구조적 충돌이 반복됐다:

| 유형 | 발생 건수 | 사례 |
|:-----|:--------:|:-----|
| Task 번호 채번 중복 | 8건 이상 | Baker 4회(TASK-B-103·121·141·159), Dave·Mike 교차 충돌(TASK-B-120·123), **TASK-B-149가 두 담당자에게 동시 배정**(Issue #552 vs #554) |
| 공유 문서(ACTIVE_TASK.md) stale 브랜치로 인한 행 삭제 | 2건 | PR#537 병합 직후, PR#538 재발 |
| 공유 물리 디렉토리 교차오염 | 2건 | Dave↔Jaison(07-17), 과거 Mike↔Dave(PR#353·367) |
| Aiden·Jaison 간 독립 채번 충돌 | 2건 | VIOLATION_TRACKER "Dave 9회" 중복 배정, IMP-136~143 번호 중복 배정 |

**공통 근본 원인**: 여러 에이전트가 "다음 번호를 스스로 확인 → 채번 → 커밋"하는 **check-then-act 경쟁 조건**을 `next-task-number.sh` 같은 셀프체크 스크립트로만 방어하고 있다. 이 방식은 동시성이 낮을 때만 통계적으로 버틴다 — 에이전트 수·커밋 빈도가 늘수록 "확인"과 "커밋" 사이 창(window)이 열려 있는 시간에 다른 에이전트가 끼어들 확률이 커진다.

**중요한 시사점**: 에이전트가 2명(Aiden·Jaison)뿐이었던 문서 채번(VIOLATION_TRACKER·IMP)에서도 정확히 같은 유형의 충돌이 발생했다 — 즉 이 문제는 "Team B 에이전트의 숙련도" 문제가 아니라 **구조 자체의 문제**다. 15명으로 늘리면 통계적으로 이 창이 훨씬 자주 열리고 닫혀, 현재의 "반려 → 재채번 → 재제출" 대응으로는 처리량 자체가 감당되지 않을 것이다.

## 3. 15인 규모에서 추가로 예상되는 문제

1. **공유 append-only 문서의 병합 충돌 폭증**: `ACTIVE_TASK.md`·`VIOLATION_TRACKER.md`·`post_launch_improvements.md`는 모든 에이전트가 파일 끝(또는 특정 섹션)에 행을 추가하는 구조 — git의 line-based diff/merge는 여러 에이전트가 "같은 근처 줄"에 동시에 쓰면 충돌로 처리한다. 15명이 병행 커밋하면 이 파일들은 사실상 상시 충돌 상태가 될 것이다.
2. **수동 CI 대체검증의 한계**: R-08-1(CI 미트리거 시 로컬 build+test 대체)은 현재 사람 2명(Aiden·Jaison)이 감당 가능한 수준이라 성립한다. 15명분 PR을 매번 수동으로 로컬 검증할 수는 없다 — CI 트리거 안정성 자체를 근본 해결해야 이 절차가 유지된다.
3. **배치 병합 주기의 처리량 한계**: 현재 "하루 1회 develop 병합" 모델은 어제 기준 diff 55개 파일·2000줄 규모였다. 15명이면 하루치 diff가 훨씬 커져 한 번에 리뷰·검증하기 어려워지고, 문제 발생 시 원인을 하루치 변경분 속에서 찾아야 하는 부담이 커진다.
4. **팀 리더(Jaison 역할) 1인 병목**: 지금 구조는 Jaison 한 명이 3명 전원의 사전검토를 담당해도 감당됐다. 15명이면 리더 1인이 모든 PR을 diff 단위로 직접 검토하는 것 자체가 병목이 된다 — 팀을 하위 그룹으로 쪼개고 하위 리더를 두는 구조가 필요해진다.

## 4. 권고안

### 4-1. 채번을 "자가할당"에서 "중앙집중 원자적 할당"으로 전환

에이전트가 커밋 전에 스스로 번호를 확인·부여하지 않는다. 대신:
- **옵션 A**: GitHub Actions가 PR 병합 시점에 번호를 자동 부여(예: 병합 커밋 훅이 `TASK-{PR번호}` 또는 원자적 카운터 발급) — Issue #358에서 이미 이 방향이 권고됐으나 미착수 상태.
- **옵션 B**: PR 번호 자체를 Task 식별자로 사용(예: `TASK-PR584`) — GitHub이 PR 번호를 원자적으로 발급하므로 경쟁 조건이 구조적으로 사라짐. 사람이 읽기엔 불편하지만 충돌 가능성 자체를 제거하는 가장 확실한 방법.

### 4-2. 공유 append-only 로그를 파일 단위로 분리

`VIOLATION_TRACKER.md`·`post_launch_improvements.md`처럼 여러 에이전트가 계속 추가하는 문서는, 팀별·일자별·에이전트별 **개별 파일**로 쪼개고 조회 시점에만 취합(스크립트 또는 대시보드)한다. 예: `violations/2026-07-18_dave.md`, `violations/2026-07-18_baker.md`. 서로 다른 파일이면 git merge 충돌 자체가 발생하지 않는다.

### 4-3. `ACTIVE_TASK.md`의 진실의 근거를 GitHub Issues/Projects로 완전 이관

손으로 관리하는 마크다운 표는 15인 동시 편집을 구조적으로 견디지 못한다. 이미 Phase 4(Issue #86)에서 "GH_ISSUES_SYNC" 자동 동기화 섹션을 도입한 선례가 있다 — 이 방향을 팀별 상세 표까지 확장해, 사람이 직접 편집하는 행을 최소화하고 GitHub Issue/Project 필드(라벨·담당자·상태)를 단일 진실 공급원으로 삼는 것을 검토한다.

### 4-4. CI 트리거 안정성 근본 해결

R-08-1(로컬 대체검증)은 2인 규모의 임시 우회책이지 15인 규모의 표준 절차가 될 수 없다. 이번 프로젝트에서 반복된 GitHub Actions `pull_request` 미트리거 현상의 재발 방지책(워크플로우 재설계, 필요 시 다른 CI 트리거 방식 검토)을 신규 프로젝트 착수 **전**에 확정해야 한다.

### 4-5. 팀 계층 구조 도입 및 병합 주기 재검토

15인을 3~5인 단위 하위 팀(각 팀 리더 1인)으로 분할하고, 하위 팀별 통합 브랜치를 두는 2단계 구조(하위팀 통합 브랜치 → 전체 통합 브랜치 → develop)를 검토한다. 배치 병합 주기도 "하루 1회"에서 "하위팀별 반나절 1회" 등으로 세분화해 diff 규모를 관리 가능한 수준으로 유지한다.

## 6. 리스크 및 트레이드오프

- 채번 자동화(4-1)는 사람이 읽기 편한 순번 체계를 일부 포기하는 대가가 있다 — 가독성보다 정합성을 우선하는 트레이드오프임을 명확히 인지해야 한다.
