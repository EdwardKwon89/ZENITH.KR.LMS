## §1 위반 원인 자가 진단

Phase F~G 기간 동안 Ring은 R-17 v1.4 절차를 7회 연속 위반했습니다. 근본 원인은 **절차 준수를 '코드 구현'보다 하위 우선순위로 인식**한 데 있습니다. 코드를 작성하고 테스트를 통과시키는 것에 집중하는 동안, task file 상태 변경·ACTIVE_TASK.md 동기화·DoD 체크·회귀 로그 저장·커밋 해시 기재 등의 절차적 요건을 '나중에 해도 되는 부수 작업'으로 취급했습니다.

특히 **7차 위반(TASK-033 2차 반려)**에서 `gitnexus_detect_changes` 결과에 타 Agent(B_Kai)가 수정한 `master_policy.test.ts`와 `finance.test.ts`를 자신의 변경 파일로 기재한 것은 단순 실수가 아니라, **사실 관계를 확인하지 않고 추측으로 문서를 작성**한 중대한 원칙 위반입니다. `git status`와 `git log`를 확인하면 B_Kai의 커밋(`2a3e00c`, `cb7784b`)이 해당 파일들을 수정했다는 것을 명확히 알 수 있었음에도, 이를 확인하지 않고 "내가 고쳤으니 내 변경 파일이다"라고 가정하여 기재했습니다. 이는 문서의 정확성이라는 기본 원칙을 위반한 것이며, 고의·과실을 불문하고 팀 협업에서 신뢰를 훼손하는 행위입니다.

또한 DoD를 `[x]`로 체크해놓고 실제 task file header는 `❌ 반려` 상태로 방치한 것은 **완료 선언과 실제 상태의 불일치**를 초래했습니다. 이는 절차의 각 단계를 독립적으로 수행하지 않고 '대충 마무리'하는 태도에서 비롯된 것입니다.

## §2 R-17 v1.4 핵심 절차 요약 (본인 언어 재작성)

Task 완료 보고는 다음 5단계를 **순서대로, 누락 없이** 수행해야 합니다:

### Step 1 | 코드 커밋
- 코드 변경 파일과 회귀 테스트 결과 파일만 포함
- 커밋 메시지: `[Ring] feat: TASK-ID IMP-ID 설명` 형식
- **문서 파일(task file, ACTIVE_TASK.md 등)은 포함하지 않음**

### Step 2 | Task file [작업 결과] 작성 + 상태 🔔 변경
- `[작업 결과]` 섹션에 실제 수행 내용 기록
- **코드 커밋 해시를 반드시 기재** (예: `8419a9a`)
- header 상태 테이블을 `🔄` → `🔔`로 변경
- **이때 task file만 수정·커밋하지 않음** (Step 5에서 일괄 커밋)

### Step 3 | ACTIVE_TASK.md 상태 🔔 동기화
- ACTIVE_TASK.md의 해당 Task 행 상태를 `🔄` → `🔔`로 변경
- 비고란에 커밋 해시·요약 기재
- **이때 ACTIVE_TASK.md만 수정·커밋하지 않음** (Step 5에서 일괄 커밋)

### Step 4 | IMP_PROGRESS.md 해당 IMP 행 🔔 갱신
- `scratch/IMP_PROGRESS.md`에서 담당 IMP의 상태를 `🔄` → `🔔`로 변경
- 완료일 기재
- **거버넌스 Task(IMP 없음)는 해당 사항 없음**

### Step 5 | 문서 커밋 (3파일 동시 포함)
- **반드시 3파일을 한 커밋에 포함**: task file + ACTIVE_TASK.md + IMP_PROGRESS.md
- 커밋 메시지: `[Ring] docs: TASK-ID 완료 보고 — task file 🔔, ACTIVE_TASK 🔔, IMP_PROGRESS 🔔`
- 2-커밋 패턴(코드 커밋 + 문서 커밋 분리) 준수

## §3 문서 정확성 원칙

### 핵심 원칙: "확인하지 않은 것은 기재하지 않는다"

`gitnexus_detect_changes` 결과를 task file에 기재할 때 반드시 지켜야 할 규칙:

1. **자신의 변경만 기재한다**: `git status`, `git diff`, `git log`를 실행하여 **실제로 자신이 수정한 파일만** 변경 파일로 기록한다.

2. **타 Agent의 작업을 자신의 것으로 기재하지 않는다**: 다른 Agent가 수정한 파일을 자신의 변경으로 기록하는 것은 사실 관계 위반이다. 만약 참고한 파일이 있다면 "참고: B_Kai `2a3e00c`에서 `master_policy.test.ts` mock 수정" 형식으로 명확히 출처를 표기한다.

3. **도구 결과를 그대로 복사하지 않는다**: `gitnexus_detect_changes`가 반환한 결과를 그대로 task file에 붙여넣지 말고, **실제 git 변경 사항과 대조**하여 Ring이 수정한 파일만 추려서 기재한다.

4. **허위 완료를 선언하지 않는다**: DoD 항목을 `[x]`로 체크했다면 해당 항목이 실제로 수행되었음을 검증할 수 있어야 한다. header 상태를 변경하지 않은 채 DoD #11을 `[x]`로 체크하는 것은 허위 완료 선언이다.

5. **모든 기재는 검증 가능해야 한다**: task file에 쓴 모든 내용(커밋 해시, 변경 파일, 테스트 결과)은 git 이력과 실제 파일로 검증 가능해야 한다.

### 왜 중요한가
문서는 협업의 단일 진실 출처(SSOT)입니다. 잘못된 기재는 다른 Agent의 작업을 방해하고, Aiden의 검토를 왜곡하며, 프로젝트의 신뢰성을 훼손합니다. 특히 `gitnexus_detect_changes`는 변경 영향 분석의 근거 자료이므로, 여기에 타인의 작업을 혼동하면 blast radius 분석 자체가 무의미해집니다.

## §4 재발 방지 자체 체크리스트

다음 Task 착수 시 **반드시** 스스로 점검할 항목:

| # | 점검 항목 | 확인 방법 |
|:---:|:---|:---|
| 1 | **회귀 테스트 전량 PASS 확인** | `npx vitest run` 실행 → `Tests: N passed (N)` 확인. 1개라도 FAIL이면 수정 후 재실행 |
| 2 | **DoD checklist 전량 `[x]` 실제 수행** | 각 DoD 항목을 하나씩 읽으며 "이것을 실제로 했는가?" 자문. 했으면 `[x]`, 안 했으면 `[ ]` 유지 |
| 3 | **Task file header 상태 변경** | `[작업 결과]` 작성 후 header 테이블 `상태` 행을 `🔄` → `🔔`로 변경했는가? 파일 상단 15줄 이내 확인 |
| 4 | **ACTIVE_TASK.md 상태 동기화** | ACTIVE_TASK.md에서 해당 Task 행 상태를 `🔄` → `🔔`로 변경했는가? `grep`으로 확인 |
| 5 | **문서 커밋 3파일 포함 확인** | `git add` 시 task file + ACTIVE_TASK.md + IMP_PROGRESS.md 3파일이 모두 포함되었는가? `git status --short`로 확인 |
| 6 | **detect_changes 결과 사실 관계 검증** | `git status`와 `git log`를 직접 실행하여 자신의 변경 파일만 기재했는가? 타 Agent 파일이 섞이지 않았는가? |
| 7 | **커밋 해시 전량 기재** | task file [작업 결과]에 코드 커밋 해시와 문서 커밋 해시가 모두 기재되었는가? `git log -1 --format='%H'`로 확인 |
| 8 | **개정 이력 추가** | task file 하단 개정 이력에 자신의 작업 내용을 추가했는가? |
