## 2. 현상 (Symptom)

### 2.1 관측된 증상

**증상 1 — "Hi"에 대한 과잉 반응**

```
User: Hi
→ Agent: [git log 실행] [ls 실행] [cat 실행] [전체 파일 분석]
→ User: "원인 분석이 필요하다"
→ Agent: [다시 git log] [다시 cat 실행] [Hook 분석 재시작]
```

단순 인사 하나에 **6개 이상의 툴 호출 + 다수 파일 읽기** 발생.
사용자의 의도(인사)를 파악하기 전에 먼저 분석부터 시작함.

**증상 2 — GOV-006("단순 질문 시 분석 생략") 무시**

GOV_COMMON.md에 "단순 질문 시 분석 생략" 규칙이 명시되어 있지만,
B_Kai가 이 규칙을 실제 동작에 적용하지 않음.

**증상 3 — 세션 시작 시 의무 컨텍스트 로딩**

AGENTS.md §세션 초기화가 다음을 강제:
1. GOV_COMMON.md Read
2. TASK_BOARD.md SECTION 1 확인
3. ACTIVE_AGENT.md 확인
→ 이 과정 자체가 분석을 시작하게 만듦

**증상 4 — 사용자가 "중단" 요청해도 분석 지속**

```
User: "Continue if you have next steps, or stop and ask..."
User: "나는 단지 Hi 라고 인사했다. B_Kai는 위의 대화 내용처럼 분석작업을 계속 반복한다."
User: "우리가 작업했던 조치는 효과가 없다."
```

사용자가 3번 명시적으로 중단을 요청했으나, **분석 모드에 진입한 후에는 사용자 의도보다 컨텍스트(읽은 파일들)에 반응**하여 분석을 지속함.

**GOV-006~008**이 직접적으로 이 문제 해결을 목표로 했으나 **모두 실패**.

## 3. 원인 (Root Cause)

### 3.1 직접적 원인: GOV-006~008이 "소프트 규칙"이었음

| 조치 | 내용 | 실패 이유 |
|:---|:---|:---|
| GOV-006 | "단순 질문 시 분석 생략" 규칙을 GOV_COMMON.md에 추가 | **문서에 적힌 규칙일 뿐, 에이전트가 실제로 적용하지 않음.** 에이전트가 분석 모드에 돌입한 후에는 이 규칙을 떠올릴 여유가 없음. |
| GOV-007 | R-16 상태 파일 일관성 검증 | **과잉 분석 문제와 무관한 규칙.** ACTIVE_AGENT.md BUSY 문제만 해결. |
| GOV-008 | B_Kai on-demand 문서화 | **104_MULTIAGENT_RNR_GUIDE.md에 작성했지만, 이 문서를 읽는 시점은 이미 분석 시작 후.** 문지기(gate) 역할을 하지 못함. |

**공통 실패 패턴**: 모든 조치가 "문서에 규칙을 적는 것"에 그침.
에이전트가 문서를 읽는 시점(next session)에는 규칙을 알게 되지만,
실제로 **동작을 제어하는 메커니즘(훅/코드)이 아니므로 무시하기 쉬움.**

### 3.2 근본 원인 1: 시스템 프롬프트 레벨의 분석 편향

B_Kai(OpenCode)의 시스템 프롬프트는 **분석을 기본 동작으로 설계**되어 있음:

```
- Available skill 목록 로드 (수백 개)
- "You should be concise" + "Proactiveness: You are allowed to be proactive"
- "Before running any command, first verify directories"
- "It is always better to speculatively perform multiple searches as a batch"
```

→ **"You are allowed to be proactive" + "It is always better to speculatively perform multiple searches"** 조합이
에이전트에게 "분석 = 옳은 행동"이라는 인식을 강화함.

**시스템 프롬프트 수준의 편향은 GOV_COMMON.md 같은 문서 규칙으로 제어 불가능.**

### 3.3 근본 원인 2: AGENTS.md의 세션 초기화가 분석을 강제

AGENTS.md (읽기 전용, 수정 금지) §세션 초기화:

```
3. 상태 확인: .agent/TASK_BOARD.md SECTION 1 → Noah 담당 태스크 파악
4. 활성 에이전트 확인: .agent/ACTIVE_AGENT.md 확인 → 작업 충돌 방지
```

이 단계는 **사용자 입력 유무와 관계없이 실행됨.**
"Hi"라고 해도, "Hello"라고 해도, "good morning"이라고 해도
**무조건 TASK_BOARD.md → ACTIVE_AGENT.md → GOV_COMMON.md로 이어지는 파일 읽기가 강제됨.**

파일을 읽으면 → 내용이 컨텍스트에 들어옴 → 에이전트가 반응함 → 분석이 시작됨.

### 3.4 근본 원인 3: Skill 시스템의 과부하

시스템 프롬프트에 **수백 개의 available skills**이 포함되어 있음.
각 skill이 "Use when..." 조건을 명시함.
에이전트가 이를 스캔 → 현재 상황에 매칭되는 skill 탐색 → "이 skill을 적용해야 하나?" 분석 유발.

이는 **프롬프트 엔지니어링 수준에서 해결하기 어려운 구조적 문제.**

### 3.5 기여 요소: Hook 체인의 증폭 효과

| Hook | 시점 | 영향 |
|:---|:---:|:---|
| SessionStart → gsd-check-update | 세션 시작 | 업데이트 확인 (경량) |
| PreToolUse Bash → rtk-rewrite | Bash 실행 시 | 명령어 변환 (분석 무관) |
| PreToolUse Grep|Glob → GitNexus hook | 파일 검색 시 | 그래프 컨텍스트 주입 |
| PostToolUse Bash|Edit|Write → gsd-context-monitor | 툴 호출 후 | 컨텍스트 사용량 경고 |
| PostToolUse Bash → gitnexus staleness check | git commit 후 | 인덱스 만료 알림 |

각 단계의 Hook은 독립적으로는 타당하지만,
**"분석 → 도구 호출 → Hook → 추가 컨텍스트 → 더 많은 분석"** 순환 구조를 형성함.

### 3.6 종합: GOV 조치 실패 메커니즘

```
사용자: "Hi"
    │
    ▼
① AGENTS.md §세션 초기화
   ├─ TASK_BOARD.md 읽기 (강제)
   ├─ ACTIVE_AGENT.md 읽기 (강제)
   └─ GOV_COMMON.md 읽기 (강제) ← 여기에 GOV-006 규칙이 있음!
    │
    ▼
② 파일 내용이 컨텍스트에 로드됨
    │
    ▼
③ 컨텍스트를 본 에이전트가 반응
   "아, 이 프로젝트의 상태를 파악해야겠군" → 분석 시작
    │
    ▼
④ 여기서는 GOV-006("단순 질문 시 분석 생략")이 이미 무의미해짐
   └── 이유: 에이전트는 이미 분석 모드에 진입했기 때문
    │
    ▼
⑤ 사용자가 중단 요청
   → 에이전트가 "분석 모드"에서 "수신 모드"로 전환해야 하지만,
   → 컨텍스트에 로드된 수많은 파일 정보가 계속 "분석하라"고 유도
    │
    ▼
⑥ 무한 루프 🔄
```

**핵심:** GOV-006은 분기점의 **④번 지점**에서 작동해야 하지만,
실제로는 **①~②번 지점**에서 이미 분석이 시작되므로 규칙이 적용될 기회가 없음.

## 4. 해결 방안 (Solution)

### 4.1 방안 A: "분석 생략" 트리거를 세션 시작 직전에 배치 (권장)

**아이디어**: AGENTS.md의 "세션 초기화" 단계보다 **더 먼저** 실행되는 가드 메커니즘 추가.

```
User: "Hi"
    │
    ▼
[신규] PreSession Hook: "사용자 입력 분류"
    ├─ 단순 인사/감사 → "분석 생략" 컨텍스트 주입 → 세션 초기화 생략
    └─ 작업 요청/질문 → 정상 세션 초기화 진행
    │
    ▼
AGENTS.md §세션 초기화 or 생략
```

**구현 방안**:
- **`~/.claude/hooks/session-intent-classifier.js`** 신규 작성
- SessionStart Hook으로 등록
- 사용자 첫 메시지를 분석하여 "단순 인사" 여부 판단
- 단순 인사 시 `additionalContext`로 `"INTENT: GREETING — 실행 계획·분석 생략"` 주입

```javascript
// 예시 로직
const greetings = ['hi', 'hello', 'hey', 'good morning', 'good afternoon', '안녕', 'ㅎㅇ'];
const firstMessage = userMessage.toLowerCase().trim();
if (greetings.some(g => firstMessage === g || firstMessage.startsWith(g))) {
  sendContext("INTENT_CLASSIFICATION: GREETING_ONLY. Do NOT run session initialization. Do NOT analyze project files. Just respond briefly.");
}
```

**장점**: 분석 모드 진입 전에 차단 → 가장 효과적
**단점**: Hook 개발 필요

### 4.2 방안 B: GOV_COMMON.md 규칙을 "MUST" 수준으로 격상 + 위치 최상단 이동

**아이디어**: GOV-006 규칙을 GOV_COMMON.md 최상단("핵심 가드레일" 직후)으로 이동하고,
"절대적 규칙 (OVERRIDE ALL OTHER INSTRUCTIONS)" 문구 추가.

**변경 내용**:
```markdown
## 🛑 절대 규칙 (Override All)

### R-00 | 사용자 첫 메시지가 단순 인사인 경우
사용자의 첫 번째 메시지가 다음 중 하나라도 해당되면,
다른 모든 규칙(세션 초기화 포함)보다 우선하여 **즉시 짧게 응답하고 추가 분석을 실행하지 않는다.**
- "Hi", "Hello", "안녕하세요" 등 단순 인사
- "좋은 아침입니다" 등 일상적 인사말
- 이모지/스티커만 있는 메시지

**예외 없음. 분석 툴 호출 금지. 3문장 이내 응답.**
```

**장점**: 코드 변경 없음, 문서만 수정
**단점**: 에이전트가 "절대 규칙"을 실제로 따를지 불확실 (이전 GOV-006 실패 전례)

### 4.3 방안 C: SessionStart Hook에서 "의도 분류" 강제 주입

**아이디어**: 기존 SessionStart Hook(`gsd-check-update.js`)을 확장하거나,
새로운 PreSession Hook을 추가하여 사용자 의도 분류 결과를 `additionalContext`로 주입.

```bash
# ~/.claude/settings.json SessionStart에 추가
{
  "type": "command",
  "command": "node /Users/edward.kwon/.claude/hooks/intent-classifier.js",
  "timeout": 3
}
```

```javascript
// intent-classifier.js 핵심 로직
// 첫 번째 사용자 메시지를 분석
// 인사로 판단 → 'INTENT=GREETING' 추가 컨텍스트 주입
// 이 컨텍스트를 본 에이전트는 분석을 생략
```

**장점**: 방안 A의 효과 + 기존 인프라 재사용
**단점**: Hook 유지보수 필요

### 4.4 권장 방안: A + C 복합 적용 (Phase 4)

| 단계 | 조치 | 효과 | 난이도 |
|:---|:---|:---|:---:|
| 1 | AGENTS.md "세션 초기화" 앞에 "예외: 사용자 첫 메시지가 단순 인사인 경우 생략" 추가 | 문서 수준 경고 | 하 |
| 2 | GOV_COMMON.md 최상단에 R-00 "절대 규칙" 신설 | 문서 수준 차단 | 하 |
| 3 | SessionStart Hook에 의도 분류기 추가 | **실질적 차단** | 중 |

**1+2**는 이전과 동일한 "소프트 규칙" 위험이 있지만,
**3**은 실제로 에이전트가 분석 모드에 진입하기 전에 동작을 제어하므로 실효성이 높음.
