## §2 R-17 v1.4 핵심 절차 요약 (본인 언어 재작성)

Task 완료 보고는 다음 5단계를 **순서대로, 분리하여** 수행해야 한다.

### Step 1 | 코드 커밋
- **내용**: 변경된 소스 코드 파일과 회귀 테스트 결과 로그 파일만 스테이징한다.
- **주의**: task file, ACTIVE_TASK.md, IMP_PROGRESS.md 등 **문서 파일은 절대 포함하지 않는다.**
- **커밋 메시지**: `[Riley] feat: IMP-XXX 설명`

### Step 2 | Task 상세 파일 작성 및 상태 변경
- **내용**: 상세 task file의 `[작업 결과]` 섹션을 작성하고, **Step 1에서 생성된 코드 커밋 해시를 명시**한다.
- **수정**: 파일 상단의 상태 메타데이터를 `🔄 구현 중` → `🔔 검토 요청`으로 변경한다.

### Step 3 | ACTIVE_TASK.md 상태 동기화
- **내용**: `.agent/ACTIVE_TASK.md` 파일에서 해당 Task 행의 상태를 `🔄` → `🔔`로 변경하고, 비고란에 커밋 해시와 간략한 작업 요약을 기재한다.

### Step 4 | IMP_PROGRESS.md 갱신
- **내용**: `scratch/IMP_PROGRESS.md`에서 담당 IMP 항목의 상태를 `🔄` → `🔔`로 갱신하고 완료일을 기록한다. (단, 거버넌스/재교육 등 IMP-ID가 부여되지 않은 공통 Task는 이 단계를 생략한다.)

### Step 5 | 문서 커밋
- **내용**: Step 2, 3, 4에서 수정한 **문서 파일(task file + ACTIVE_TASK.md + IMP_PROGRESS.md)을 한데 묶어 하나의 커밋으로 제출**한다.
- **커밋 메시지**: `[Riley] docs: TASK-XXX 완료 보고 — task file 🔔`
- **종료 작업**: 문서 커밋이 완료되면, 최종적으로 생성된 문서 커밋 해시를 task file의 `[작업 결과]`에 업데이트하여 최종 완료한다. (Aiden이 최종 승인하기 전까지는 절대로 `✅` 상태로 변경하지 않는다.)

## §3 착수 절차 및 파일 조작 원칙

### 1. 설계 확정(🔍) 전 구현 금지 원칙
- **이유**: 에이전트가 독단적으로 구현을 시작할 경우, 상위 설계 사양이나 요구사항(예: 다중 경로 정산 방식, RLS 정책 등)과 다른 방향으로 코딩이 진행될 수 있다. 이는 사후 수정에 따른 불필요한 공수 낭비와 타 기능과의 호환성 문제를 유발하므로, 구현 전에 합의된 사양을 명문화하여 승인(🔍→🔄)을 받는 절차가 필수적이다.
- **올바른 기준**: 복잡도가 지극히 낮은 단순 Task를 제외하고, 복수 대안 검토나 데이터 스키마 수정이 따르는 모든 작업은 반드시 `[설계 의견]` 섹션을 제출(📝)하고, Aiden의 `[설계 확정]`(🔍→🔄)을 득한 후에만 소스 코드 작성을 시작해야 한다.

### 2. 타 에이전트 파일 조작 및 크로스 커밋 금지 원칙
- **이유**: 각 에이전트의 상세 task file은 해당 에이전트의 책임 영역이자 독립적인 증적이다. 타 에이전트의 파일을 자신의 커밋에 포함시키거나 무단 수정하는 행위는 Git 충돌의 원인이 되며, 작업 이력 추적과 검증 과정의 단일 진실 원칙(SSOT)을 무너뜨린다.
- **올바른 기준**: `git add .` 대신 반드시 `git add path/to/file`과 같이 작업한 소스 파일과 문서 파일을 명시적으로 지정하여 커밋을 구성해야 한다. 커밋 전에 `git status`로 스테이징 목록을 확인하여 자신에게 할당되지 않은 파일(특히 타 Agent의 task file)이 섞여 있는지 엄밀히 검증한다.
