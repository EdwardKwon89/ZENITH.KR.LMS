### 1. 요구사항 명확화

```

□ 사용자의 목표는 무엇인가?
  답변: _________________________________
  
□ 성공 기준은 무엇인가?
  답변: _________________________________
  - [ ] 기준 1: _______________
  - [ ] 기준 2: _______________
  - [ ] 기준 3: _______________

  □ 없음
  □ 있음: ___________________________
```

**체크:** 모든 항목이 명확한가? → YES ✓ / NO ✗

### 2. 아키텍처 설계

```
□ 이 기능을 구현하려면 어떤 파일을 수정해야 하나?
  
  파일 1: _________________ (역할: _____________)
  파일 2: _________________ (역할: _____________)
  파일 3: _________________ (역할: _____________)
  [필요하면 추가]

□ 주요 함수/메소드는 무엇인가?

  함수 1: _________________
    입력: _______________
    출력: _______________
    역할: _______________
    
  함수 2: _________________
    입력: _______________
    출력: _______________
    역할: _______________
    
  [필요하면 추가]

□ 데이터 흐름을 이해하고 있는가?
  
  입력 데이터: _______________
  처리 로직: __________________
  출력 데이터: _______________
  
  흐름도:
  ┌─────────┐   ┌─────────┐   ┌─────────┐
  │입력데이터│→ │처리로직 │→ │출력데이터│
  └─────────┘   └─────────┘   └─────────┘

□ 기존 코드와의 통합 계획이 있는가?
  □ 새 파일 (완전 신규)
  □ 기존 파일 수정 (어느 부분인지 명확한가?)
  □ 기존 함수 호출 (호출 방식이 명확한가?)
  
  기존 코드 참고:
  - [ ] 관련 파일 읽음
  - [ ] 네이밍 규칙 확인
  - [ ] API/함수 시그니처 확인
```

**체크:** 아키텍처가 명확한가? → YES ✓ / NO ✗

### 3. 의존성 확인

```
□ 외부 라이브러리 필요한가?
  □ 필요 없음
  □ 필요함:
    - [ ] 라이브러리명: ______________ (사유: _______)
    - [ ] 라이브러리명: ______________ (사유: _______)
    
  확인:
    - [ ] 이미 설치되어 있는가?
    - [ ] 버전 호환성은 OK인가?
    - [ ] 보안 이슈는 없는가?

□ 다른 팀원과의 의존성이 있는가?
  □ 없음
  □ 있음:
    - [ ] 담당자: ____________ (기능: __________)
    - [ ] 완료 예정일: __________
    - [ ] 조율됨: YES / NO
    
  조율 방법:
  - [ ] Slack에서 확인
  - [ ] 개발 계획 공유됨
  - [ ] 예상 완료일 기록

□ 데이터베이스 변경 필요한가?
  □ 없음
  □ 필요함:
    - [ ] 테이블: ___________
    - [ ] 수정 내용: __________
    - [ ] Migration 계획: ______

□ Hierarchy Check: 조직, 장소 등 관계형 데이터 설계 시 계층(본사-지사) 구조 필요성 확인 (SAR-002) [2026-04-17 추가]
□ Entity Analysis: 도메인 엔티티의 "그룹화(Grouping)" 및 "상속" 관계 여부 확인 (SAR-002) [2026-04-17 추가]
```

**체크:** 의존성이 해결되었는가? → YES ✓ / NO ✗

### 4. 예상 이슈 파악

```
□ 예상되는 기술적 문제가 있는가?
  
  예상 이슈 1: ________________________
  └─ 심각도: [ ] 높음 [ ] 중간 [ ] 낮음
  └─ 해결책: ________________________
  
  예상 이슈 2: ________________________
  └─ 심각도: [ ] 높음 [ ] 중간 [ ] 낮음
  └─ 해결책: ________________________
  
  예상 이슈 3: ________________________
  └─ 심각도: [ ] 높음 [ ] 중간 [ ] 낮음
  └─ 해결책: ________________________

□ 성능/보안 고려사항이 있는가?
  
  성능:
  □ N+1 쿼리 문제 없는가?
  □ 대용량 데이터 처리 계획이 있는가?
  □ 캐싱 필요한가?
  
  보안:
  □ 사용자 입력 검증 계획이 있는가?
  □ SQL Injection/XSS 위험 없는가?
  □ 권한 검증 필요한가?

□ 테스트 불가능한 부분이 있는가?
  □ 없음
  □ 있음:
    - [ ] 부분: ________________
    - [ ] 이유: ________________
    - [ ] 해결책: ________________
```

**체크:** 예상 이슈가 파악되고 대책이 있는가? → YES ✓ / NO ✗

### 6. 문서 검증

```
□ 관련 설계 문서가 있는가?
  □ 없음 (신규 기능이므로 새로 작성할 필요 없음)
  □ 있음:
    - [ ] 문서 경로: ________________
    - [ ] 링크 정상: YES / NO
    - [ ] 내용 최신: YES / NO

□ README 또는 기술 문서의 링크가 실제 파일과 일치하는가?
  □ 문서 링크 검증 필요 없음 (문서 변경 없음)
  □ 필요함:
    - [ ] 기존 README 파일명 변경 여부 확인
    - [ ] 변경 시 모든 링크 갱신 여부 확인
    - [ ] 링크 정상 작동 확인 (클릭 테스트)
    
  참고: SAR-001 (문서 링크 오류 예방)

□ 새 파일 추가 시 인덱스 문서 업데이트 계획이 있는가?
  □ 새 파일 추가 없음
  □ 있음:
    - [ ] 해당 폴더의 README 파일 확인
    - [ ] README에 새 파일 목록 추가 필요한가?
    - [ ] 링크 추가 여부 확인

□ Cross-doc Sync: 주요 비즈니스 정책 변경 시 관련 파일(An, Fun, WBS) 간 명칭/번호 정합성 확인 (SAR-002) [2026-04-17 추가]
```

**체크:** 문서가 완전하고 링크가 정상인가? → YES ✓ / NO ✗
