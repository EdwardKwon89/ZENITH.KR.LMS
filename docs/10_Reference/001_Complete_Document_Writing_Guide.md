## 문서 분류 시스템

### 1. 문서 타입별 번호 체계

```
파일 번호: [세자리숫자]_[파일명].md

100-199: 계획 & 분석 (Planning & Analysis)
├─ 101: 요약/개요 (Executive Summary)
├─ 102: 작업 구조 (WBS, Project Structure)
├─ 103: 기준선 (Baseline, Reference)
├─ 104: 현황 분석 (Current State Analysis)
└─ 10X: 기타 분석 문서

200-299: 기술 설계 (Technical Design)
├─ 201: 기술 스택 (Technology Stack)
├─ 202: 아키텍처 설계 (Architecture Design)
├─ 203: 데이터 모델 (Data Model)
└─ 20X: 기타 기술 문서

300-399: 구현 전략 (Implementation Strategy)
├─ 301: 구현 전략 (Implementation Plan)
├─ 302: 개발 표준 (Development Standards)
└─ 30X: 기타 구현 문서

400-599: 운영 & 관리 (Operations & Management)
├─ 401: 일정 및 리스크 (Timeline & Risk)
├─ 501-599: 품질, 모니터링, 운영 관련
└─ 4XX: 기타 운영 문서

001-099: 분석 결과 (Analysis Results) - BaseLine
├─ 001-010: 시스템별 분석
├─ 011-099: 통합 분석, 기타
└─ 000: README 인덱스 (각 폴더별)
```

### 2. 분류 이모지 시스템

```
📌 계획 & 분석      (Planning & Analysis)

🚀 구현 전략       (Implementation Strategy)
⏱️ 운영 & 관리     (Operations & Management)
🔍 분석 결과       (Analysis Results)
```

## 폴더 및 파일 명명 규칙

### 1. 폴더 명명 규칙

#### 프로젝트 계획 폴더
```
MasterPlan/                          # 프로젝트 전체 전략 및 계획
├── 101_Executive_Summary.md
├── 102_Project_WBS.md
└── ...
```

#### 분석 폴더 (분류별)
```
BaseLine(As-Is)/                     # 기존 시스템 분석
├── 01_Plaza_Operation_App/          # [번호]_[영문_시스템명]
├── 02_Toll_Center/
├── 03_Clearing_Center/
├── 04_IF_App/
├── 05_Integrated_Analysis/          # 통합 분석
└── 06_Reference_System/             # 참조 시스템
```

**폴더 명명 패턴**:
- 형식: `[두자리번호]_[영문_폴더명]`
- 규칙:
  - 01, 02, 03... 순서대로 번호 부여
  - 영문만 사용 (한글 불가)
  - 공백은 언더스코어(_) 사용
  - 예: `05_Integrated_Analysis`, `06_Reference_System`

### 2. 파일 명명 규칙

#### 프로젝트 계획 문서
```
[세자리번호]_[파일명].md

예:
- 101_Executive_Summary.md
- 102_Project_WBS.md
- 504_Quality_Audit_Strategy.md
```

#### 분석 결과 문서
```
[세자리번호]_[파일명].md

예:
- 000_README.md              # 폴더 인덱스
- 001_Overview.md            # 시스템 개요
- 002_Features_Tree.md       # 기능 구조
- 003_User_Roles.md          # 사용자 역할
- 004_Business_Workflows.md  # 비즈니스 흐름
```

**규칙**:
- 세자리 숫자: 000-099 (분석), 100-599 (계획)
- 파일명: 영문 케밥케이스 (kebab-case)
  - ✅ 올바른 예: `002_Features_Tree.md`
  - ❌ 잘못된 예: `002_FeaturesTree.md`, `002_Features_tree.md`
- 한글 사용 가능하나 영문 권장
  - 예: `001_권한_및_역할_매트릭스.md` (허용)
- 특수 문자 제외: `-`, `_` 만 사용

## 문서 메타데이터 표준

### 1. 메타데이터 위치 및 형식

#### 프로젝트 계획 문서 (MasterPlan)
```markdown
---
### [문서 정보]
- **버전**: v1.0
- **작성일**: 2026년 3월 26일
- **최종 수정**: 2026년 3월 26일
- **작성자**: Antigravity (AI)
- **개요/변경사항**: 초안 작성 / 변경 사항 요약
---
```

**위치**: 파일 최상단 (네비게이션 이전)
**필수 항목**:
- 버전 (v버전번호)
- 작성일 (YYYY년 M월 D일 형식)
- 최종 수정 (초기: 작성일과 동일, 이후 수정시 업데이트)
- 작성자 (역할/이름)
- 개요/변경사항 (간단한 한 줄 요약)

#### 분석 결과 문서 (BaseLine)
```markdown
## 📌 메타데이터
- **분석 대상**: `Plaza Operation App. User Manual.pdf`
- **대상 페이지**: 1~50페이지
- **작성일**: 2026-03-24
- **생성자**: Claude Code
```

**위치**: 본문 H2 섹션으로 포함
**필수 항목**:
- 분석 대상 (원본 파일명)
- 대상 페이지 또는 범위
- 작성일 (YYYY-MM-DD 형식)
- 생성자

### 2. 문서 구조 메타데이터

#### README 헤더 (폴더 인덱스용)

## 콘텐츠 포맷 규칙

### 1. 제목 구조 (Heading)

```markdown
# H1: 문서 제목 (문서 번호. 문서명)
## H2: 주요 섹션 (이모지 + 섹션명)
### H3: 하위 섹션
#### H4: 상세 섹션
```

**규칙**:
- H1: 한 번만 사용, 문서의 메인 제목
- H2부터: 이모지 + 섹션명
- 계층 구조: H1 → H2 → H3 → H4 (최대 4단계)

#### 예시
```markdown
# 102. 프로젝트 WBS (Project Work Breakdown Structure)

## 📋 개요

### WBS 설계 원칙

#### 완전성 원칙
```

### 2. 이모지 사용 규칙

#### 섹션 헤더용 이모지
```
📋 개요, 목차, 내용
🏗️ 구조, 설계, 아키텍처
🛠️ 기술, 도구, 구현
🚀 배포, 전략, 계획
⏱️ 일정, 시간, 관리
🎯 목표, 요약, 핵심
🔍 분석, 검토, 상세
📊 데이터, 메트릭, 통계
📌 정보, 메타데이터, 주의
🔗 링크, 참조, 관계
🔄 프로세스, 순환, 업데이트
✅ 완료, 성공, 긍정
❌ 미완료, 실패, 부정
⭐ 신규, 중요, 강조
🟡 경고, 주의

```

#### 인라인 이모지
```

❌ 미완료/부정
⭐ 신규/중요
🟡 경고/주의
🔴 에러
💚 성공
🟠 주의
💙 정보
```

### 3. 테이블 포맷

```markdown
| 헤더1 | 헤더2 | 헤더3 |
|-------|-------|-------|
| 항목1 | 값1 | 설명1 |
| 항목2 | 값2 | 설명2 |
```

**규칙**:
- 헤더 행: 굵음 (자동)
- 구분선: `|---|---|---|` (최소 3개 `-`)
- 정렬: 기본(좌측), 중앙(`:-|`), 우측(`-:|`)
- 너비: 콘텐츠에 맞춤

### 4. 코드 블록

```markdown
# 다이어그램 (텍스트 기반)
├─ Level 1
│  ├─ Level 2
│  └─ Level 2-2

# 설정/구성 파일
---
# YAML 형식
key: value
```

**규칙**:
- 언어 지정: ` ```markdown `, ` ```yaml `, ` ```bash `
- Tree 구조: `├─`, `│`, `└─` 문자 사용
- 길이 제한: 80자 이내 권장

### 5. 리스트 포맷

```markdown
# Unordered List (순서 없음)
- 항목 1
- 항목 2
  - 하위 항목 2-1
  - 하위 항목 2-2

# Ordered List (순서 있음)
1. 첫 번째
2. 두 번째
   1. 2-1 항목
   2. 2-2 항목
```
