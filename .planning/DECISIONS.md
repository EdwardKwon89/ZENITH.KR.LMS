### 4. Self_Audit_Report (SAR) 도입 (2026-04-08)

**컨텍스트:**
- Self Check & Self Test에서 발견된 오류에 대한 체계적 기록 필요
- 동일 오류의 재발 방지 메커니즘 부재
- 팀 학습 및 프로세스 개선을 위한 자산 부족

**결정:**

- **Self_Audit_Report (SAR)** 체계 도입
- Self Check/Self Test 미통과 시 필수로 SAR 작성
- SAR 작성 후 Check List 업데이트로 재발 방지

**근거:**

1. **오류 추적성**
   - 모든 오류가 기록되고 추적 가능
   - 근본 원인 분석 가능
   - 패턴 식별 가능

2. **재발 방지**
   - Check List 업데이트로 다음 작업에서 동일 오류 방지
   - 예방적 조치 추가
   - 조직 학습 축적

3. **팀 학습**
   - 모든 팀원이 오류 경험 학습
   - 베스트 프랙티스 공유
   - 품질 문화 강화

4. **프로세스 개선**
   - SAR 분석으로 프로세스 문제 식별
   - 주기적 개선 가능
   - 근본 원인 해결

### 5. 미들웨어 세션 안정성 및 쿠키 무결성 가드레일 (2026-04-18)

**컨텍스트:**
- Next.js 미들웨어에서 `NextResponse.next()` 호출 시 병합되는 쿠키의 속성(Path, HttpOnly 등)이 유실되어 인증 무한 루프가 발생하는 현상 발견.
- 로컬 개발 환경과 배포 환경 간의 속성 불일치 문제 해결 필요.

**결정:**
- **Header-based Cookie Fusion**: `set-cookie` 헤더를 직접 파싱하고 병합하여 쿠키의 모든 메타데이터를 100% 보존하는 유틸리티 도입.
- **Fail-safe Redirect**: 리다이렉트 발생 시 반드시 해당 Locale의 Base Path로 수렴하도록 강제.

**근거:**
- `NextResponse.cookies` API의 한계(속성 손실 가능성) 극복.
- 세션 스티키니스(Session Stickiness) 보장 및 무한 루프 원천 차단.

**영향범위:**
- `middleware.ts`, 인증 유틸리티 전반.
- 향후 모든 쿠키 수동 조작 로직.

**상태:** 완료 (SAR-2026-04-18-001)

### 8. 사용자 소속 기반 오더 등록 정책 및 화주 자동 할당 (2026-04-21)

**컨텍스트:**
- 오더 등록 시 화주(Shipper)를 수동으로 선택함에 따른 데이터 오입력 및 타 법인 명의 도용 위험 존재.
- 법인 사용자와 개인 사용자의 비즈니스 프로세스(B2B vs B2C)를 명확히 분리하여 전문성 확보 필요.

**결정:**
- **Affiliation-based Automation**: 로그인한 사용자의 `org_id` 유무에 따라 오더 등록 환경을 자동 분기함.
- **Corporate Account (B2B Only)**: 소속 법인이 있는 경우 'B2B 일반 화물'만 등록 가능하며, 화주는 소속 법인으로 강제 고정함.
- **Individual Account (B2C Only)**: 소속 법인이 없는 경우 B2B 탭을 비활성화하고, 시스템이 제공하는 '개인용 더미 화주'를 자동 할당하여 B2C 오더만 허용함.
- **Enforcement Level**: 클라이언트 UI 제약뿐만 아니라 서버 액션 레이어(`validateUserAction`)에서도 소속 정보의 정합성을 검증하도록 함.

**근거:**
- 법인 고객의 데이터 보안 및 무결성 강화.
- 개인 사용자에게 불필요한 복잡한 B2B 옵션 노출 차단.
- 플랫폼 운영 효율성 및 휴먼 에러 방지.

**영향범위:**
- 오더 등록 UI (`OrderRegistrationForm.tsx`), 마스터 데이터 API, 오더 저장 서버 액션.

**상태:** 완료

### 9. 실시간 물류 요율 엔진 및 인프라 노드 제약 정책 (2026-04-21)

**컨텍스트:**
- 오더 등록 시 화주에게 즉각적인 비용 통찰력을 제공하여 의사결정을 지원해야 함.
- 항공, 해상 등 운송 수단별로 상이한 운임 산출 기준(Chargeable Weight)의 표준화 필요.
- 운송 수단과 무관한 항구/공항 선택으로 인한 데이터 정합성 오류 방지 필요.

**결정:**
- **Standardized Chargeable Weight Engine**:
  - **항공/특송 (AIR/EXP)**: L*W*H / 6,000 기준($167kg/CBM$) 부피 중량과 실중량 중 큰 값 채택.
  - **해상 (SEA)**: 1,000kg 또는 1CBM 중 큰 값을 **Revenue Ton (R.T)**으로 산출.
- **Dynamic Infrastructure Filtering**:
  - `transport_mode` 선택에 따라 `origin_port_id`, `dest_port_id`의 선택 가능 목록을 실시간 필터링 (e.g. SEA 선택 시 Airport 노출 차단).
  - 모드 변경 시 기존 선택된 노드 데이터를 자동 초기화하여 정합성 강제.

**근거:**
- 물류 업계 표준 산출 방식 준수를 통한 대외 신뢰도 확보.
- 사용자 입력 단계에서의 오류 방지를 통한 운영 비용 절감.
- 실시간 견적 제공을 통한 고객 경험(UX) 차별화.

**영향범위:**
- 오더 등록 UI (`OrderRegistrationForm.tsx`), 물류 유틸리티 (`freight-calculator.ts`), 오더 저장 스키마.

**상태:** 완료 (v2.2)

### 10. GS1-128 (SSCC-18) 기반 마스터 오더 번호 표준화 (2026-04-21)

**컨텍스트:**
- 글로벌 물류 가시성 확보와 하드웨어 스캔 호환성을 위해 국내외 창고에서 통용되는 식별 체계 필요.
- 기존 임의 문자열 방식(`MO-...`)은 자동화 설비에서의 인식률이 낮음.

**결정:**
- **SSCC-18 (Serial Shipping Container Code)** 채택:
  - AI(00) + Extension(1) + 제니스 코드(88012345) + 일련번호(8자리) + Check Digit(1자리).
  - 총 18자리의 숫자로 구성된 글로벌 고유 식별자 활용.
- **Luhn Mod 10 Checksum**: 데이터 정합성 보장을 위한 체크 디지트 알고리즘 강제.

**근거:**
- 국제 물류 바코드 표준(GS1-128) 준수.
- 스캔 오류 발생 시 시스템 레벨에서의 즉각적인 유효성 검증 가능.

**영향범위:**
- `generateMasterOrderNo` 유틸리티, 마스터 오더 DB 스키마, 라벨 인쇄 콤포넌트.

**상태:** 완료
